# GeneNetworks Algorithm And Performance Review

Date: 2026-05-24

Scope:

- Compare `https://sysndd.dbmr.unibe.ch/GeneNetworks` with local `http://localhost:5173/GeneNetworks`.
- Review the functional clustering and network rendering algorithm.
- Explain why apparently tiny clusters, including 3-gene views, can appear.
- Identify stability and performance improvements.
- Ground recommendations in code evidence, runtime measurements, and current external documentation.

This is a read-only technical review. No runtime code was changed.

## Executive Summary

The current GeneNetworks page has two different performance problems:

1. Cold API/cache computation is slow and blocks first useful render. In this session, a local cold `GET /api/analysis/network_edges?cluster_type=clusters&max_edges=10000` took 55.1s by curl and about 20.0s in a Playwright run after partial warming. The immediate warm repeat took 0.58s. Production was already warm and returned the same endpoint in about 1.0-1.5s.
2. The browser still spends 3-5s on a main-thread Cytoscape `fcose` layout even though the API already computes server-side layout coordinates. The frontend drops `node.x` and `node.y`, creates elements without `position`, and runs a fresh `fcose` layout with `numIter: 2500`.

The apparent "clusters with 3 genes" are not true backend functional clusters in the default endpoint. The default R clustering path enforces `min_size = 10`. The observed production case is a display subset: production cluster 10 has 13 genes in the table, 7 nodes after the network edge cap, and only 3 visible genes under the default "Definitive only" category filter.

Production and local diverge for expected reasons:

- Local API/app version is `0.20.10`; production is `0.20.6`.
- The input data differs: local has 3,214 NDD phenotype genes; production has 3,242.
- The resulting cached Leiden clustering differs: local has 8 main clusters; production has 10.
- Cache warmth differs: production was warm; local was cold after the dev stack restart and then warm.

The highest-return fixes are:

1. Preserve the current fCoSE visual grammar by precomputing and caching Cytoscape/fCoSE display coordinates off the browser main thread, then render them with Cytoscape `preset`. The existing API `x/y` coordinates are a useful fast fallback/proof, but they are generated by igraph DrL/FR and should not be treated as the final visual-preserving layout.
2. Stop full cluster result loading through public async job status on first page load. Use a minimal or paginated cluster endpoint for initial table state, then load enrichments/details on selection.
3. Do not trigger LLM batch generation from public page cache-hit requests. Trigger it only from admin/prewarm workflows or when a new cluster hash appears.
4. Make network cache keys data-aware. `gen_network_edges_mem()` is keyed only by `cluster_type` and `min_confidence`, even though it reads database state internally.
5. Preserve and label true cluster sizes separately from retained network nodes and visible filtered nodes.

## Current Algorithm

Functional clustering:

1. Query distinct NDD genes from `ndd_entity_view` where `ndd_phenotype == 1`.
2. Join to `non_alt_loci_set` genes with non-null `STRING_id`.
3. Build an induced subgraph from STRINGdb v11.5 at score threshold 400 by default.
4. Run either Walktrap or Leiden. The default is Leiden:
   - `set.seed(42)`.
   - `objective_function = "modularity"`.
   - `resolution_parameter = 1.0`.
   - `beta = 0.01`.
   - `n_iterations = 2`.
5. Convert memberships to clusters.
6. Hash cluster symbols through `post_db_hash()`.
7. Filter clusters with `cluster_size >= min_size`, with default `min_size = 10`.
8. Run STRING enrichment per retained cluster.
9. Recursively compute subclusters, also inheriting `min_size = 10`.

Network edge endpoint:

1. Recomputes or reads the functional cluster object through `gen_string_clust_obj_mem()`.
2. Maps cluster memberships to HGNC IDs.
3. Reads STRING graph at requested `min_confidence`.
4. Builds an induced subgraph for clustered genes.
5. Converts STRING protein IDs to HGNC IDs.
6. Computes server-side layout coordinates with igraph:
   - DrL for more than 1,000 nodes.
   - FR-grid for 501-1,000 nodes.
   - FR for up to 500 nodes.
7. Returns nodes with `x/y`, edges, and metadata.
8. Applies `max_edges` after full network generation; current UI requests 10,000 edges.

Important implementation details:

- `network_edges` accepts `min_confidence`, but the nested functional clustering call in `analysis-network-functions.R:151` does not pass `score_threshold = min_confidence`. That means the edge graph may use the requested threshold while cluster assignments remain based on the clustering default.
- The code does not explicitly pass `combined_score` as Leiden weights. If the STRINGdb graph lacks a standard `weight` edge attribute, community detection treats all retained interactions equally.
- Duplicate STRING IDs are collapsed to the first HGNC ID in `analysis-network-functions.R:204`, so network node count is not the same as clustered gene count.
- The frontend type already has optional `x` and `y` on `NetworkNode`, but `useNetworkData.ts` does not convert them into Cytoscape element `position`.

## Why A 3-Gene Cluster Appears

The default backend functional clustering path should not return a true 3-gene cluster. The invariant is:

- `gen_string_clust_obj(min_size = 10)` by default.
- Returned main clusters are filtered by `cluster_size >= min_size`.
- Recursive subclusters inherit the same default.

The production visual case is different:

1. The true table cluster has 13 genes.
2. The network endpoint first caps edges to the top 10,000 globally and then drops nodes not connected by retained edges.
3. The network UI defaults to `Definitive only`, hiding Limited and Moderate genes.
4. The tooltip counts visible children, not true cluster size.

So the visible 3-gene group is valid as a filtered display subset but misleading if interpreted as a true functional cluster.

This distinction should be explicit in both metadata and UI copy:

- `cluster_size`: all genes in the algorithmic cluster.
- `retained_network_nodes`: nodes still present after the edge cap.
- `visible_nodes`: nodes visible after UI category/cluster filters.

## Performance Root Causes

### 1. Server Layout Is Computed Then Ignored

`gen_network_edges()` computes `x/y` positions and reports `layout_algorithm` and `layout_time_seconds`. The frontend currently discards those positions and runs `fcose` in the browser.

Evidence:

- API node records include `x` and `y`.
- `app/src/types/models.ts:182` declares `x?: number` and `y?: number`.
- `app/src/composables/useNetworkData.ts:173` builds Cytoscape nodes without a `position` field.
- `app/src/composables/useCytoscape.ts:507` runs `fcose` with `numIter: 2500`.
- Playwright recorded 3-5s main-thread long tasks matching this layout.

This is the clearest immediate waste. However, current server coordinates come from igraph DrL/FR, while the current user-visible geometry comes from Cytoscape fCoSE over a compound graph. If the UI representation must stay visually the same, the final fix should precompute fCoSE coordinates server-side or worker-side, not simply switch the browser to the current igraph coordinates.

### Focused Addendum: Preserve fCoSE While Removing The Main-Thread Stall

User constraint: the GeneNetworks UI representation should stay the same. In practice, that means keeping the current Cytoscape/fCoSE compound graph layout semantics:

- cluster parent compound nodes,
- gene child nodes with the same `size` and cluster color fields,
- the same displayed edge set after `max_edges`,
- the same fCoSE layout profile and package versions,
- the same Cytoscape styling, interactions, filters, tooltips, export behavior, and table synchronization.

The current API coordinates do not meet that visual-sameness constraint because they are computed before `max_edges` limiting and by a different algorithm:

- API: igraph DrL/FR coordinates on the full induced STRING graph, normalized to `0-1000`.
- Browser: Cytoscape fCoSE coordinates on the transformed compound Cytoscape element graph, after the endpoint has limited the displayed edge set.

Local headless Cytoscape/fCoSE prototype, using the captured local `network_edges` payload:

| Prototype path | Payload | Layout time |
| --- | ---: | ---: |
| Cytoscape `preset` using current API DrL `x/y` | 2,259 nodes / 10,000 edges | 9 ms |
| Current browser-style fCoSE options, randomized | 2,259 nodes / 10,000 edges | 3,105 ms |
| Current fCoSE initialized from API DrL coordinates | 2,259 nodes / 10,000 edges | 2,521 ms |
| fCoSE `draft` quality | 2,259 nodes / 10,000 edges | 584 ms |
| Cytoscape `preset` using current API DrL `x/y` | 1,439 nodes / 3,000 edges | 18 ms |
| Current browser-style fCoSE options, randomized | 1,439 nodes / 3,000 edges | 4,981 ms |

Interpretation:

- `preset` rendering is effectively free compared with fCoSE and is the right browser path.
- Initializing fCoSE from API DrL positions helps only modestly and still leaves a multi-second layout.
- `draft` fCoSE is faster but changes quality/geometry, so it is a fallback/performance profile, not a visual-preserving default.
- The visual-preserving default should be: compute fCoSE once outside the browser, store coordinates, and let the browser use `preset`.

Recommended API/worker design:

1. Keep `gen_network_edges_mem()` as the base biological graph artifact: nodes, edges, category, cluster assignment, and existing igraph coordinates for fallback/debug.
2. Add a separate display layout artifact, e.g. `gene_network_cytoscape_layout`, computed from the exact Cytoscape element graph the frontend currently builds.
3. Compute that artifact in a worker/admin prewarm path, not in public request handling.
4. Use Node/Cytoscape/fCoSE for the layout helper, either inside the worker image or as a small internal layout sidecar. The current API image is R-only, while Cytoscape/fCoSE already live in the frontend package.
5. Include `max_edges` and the edge filter policy in the display layout key. Layout positions must correspond to the displayed edge set, not the full 68k-edge graph.
6. Return fCoSE-derived positions either as `nodes[].x/y` with `metadata.layout_engine = "cytoscape-fcose"` or as a separate `layout.positions` map keyed by Cytoscape node ID.
7. Frontend should add `position` to gene node elements when complete positions are available and choose `layout: { name: "preset" }`. If any gene position is missing or invalid, it should keep the current fCoSE fallback.
8. `resetLayout()` should restore the stored preset positions when a layout artifact exists. A separate explicit "recompute layout" action can still run client fCoSE for diagnostics or ad-hoc exploration.

Recommended display-layout cache key dimensions:

- artifact type and schema version,
- layout version and layout profile name,
- `layout_engine = cytoscape-fcose`,
- Cytoscape.js version, cytoscape-fcose version, and Node/runtime version,
- `cluster_type`, `min_confidence`, `max_edges`,
- deterministic edge filter policy and tie-breakers,
- STRING version,
- sorted NDD gene-set hash,
- STRING ID mapping hash,
- cluster membership hash,
- exact displayed edge-set hash,
- node shape/style-affecting hash for size and compound-parent policy,
- fCoSE options hash.

Determinism caveat:

- Current browser fCoSE uses `randomize: true`, so pixel-identical layouts are not a stable contract today.
- The fCoSE documentation states that `randomize: false` requires `quality: "proof"`, which may be slower. A production artifact can still be stable for users by storing the computed randomized layout once and serving it until the input/layout hash changes.
- If strict reproducibility across rebuilds is required, test `quality: "proof"` with deterministic initial positions, but do not assume it is cheaper than cached randomized default fCoSE.

This design follows Cytoscape.js guidance: layouts map nodes to positions; `preset` is the intended way to use predetermined coordinates; and headless/precomputed layout is a documented performance strategy when client-side layout is too slow. It also preserves the current fCoSE compound-graph representation instead of silently changing the visual model.

### 2. First Request Can Still Do Heavy Work In The Request Path

Both synchronous endpoints can trigger heavy R work on cache miss:

- `functional_clustering` calls `gen_string_clust_obj_mem()` in-request.
- `network_edges` calls `gen_network_edges_mem()` in-request.

`gen_network_edges_mem()` internally may require:

- STRING graph access.
- functional cluster object.
- enrichment in the nested clustering object.
- edge extraction.
- server layout.
- RDS serialization.

The warm path is acceptable, but the cold path is not acceptable for a user-facing route.

### 3. The Page Loads A Huge Cluster Result Up Front

The normal page path submits a clustering job and then pulls the full result through job status. This includes nested identifiers, term enrichment, and subclusters. In this review, the status result was about 11 MB decoded.

The synchronous endpoint already supports pagination, but the main page path does not use that as the initial table contract.

### 4. Cache-Hit Public Requests Trigger LLM Batch Work

On cache hit, `jobs_endpoints.R:165` triggers `trigger_llm_batch_generation()` for functional clusters. In the local logs, loading the public page created an LLM generation job that iterated over all 8 cached functional clusters and checked cached summaries.

Even when summaries are all cached, this adds DB writes, worker queue noise, and background load to a public read path. LLM generation is an admin/prewarm concern, not a public page-load side effect.

### 5. Cache Keys Are Not Data-Aware Enough

`gen_string_clust_obj_mem()` is keyed by the HGNC list and explicit arguments, so it changes when the NDD gene list changes. It still does not directly include:

- `non_alt_loci_set` mapping version.
- STRING local data version beyond code constants.
- clustering implementation version beyond memoise internals.
- relevant status/category updates that do not change the gene list.

`gen_network_edges_mem()` is more fragile: it is called with only `cluster_type` and `min_confidence`, but the function reads database state internally. A database change can leave the network cache stale until TTL expiry or `CACHE_VERSION` invalidation.

The disk cache has `max_age = 86400`, so even healthy caches can become cold once per day.

## Stability Risks

### Cluster Numbers Are Not Stable Semantic Identifiers

Leiden membership numbers can shift when input data changes. The cluster hash by composition is a better stable identity for summaries and references. UI should avoid implying that "Cluster 10" is durable across releases.

### Medium-Confidence Edges Add Sensitivity

STRING score 400 is medium confidence. STRING documentation frames scores as confidence/probability-like estimates and notes that higher thresholds produce smaller, clearer networks. For a default public interpretive view, a high-confidence core plus optional medium-confidence expansion is more stable than treating 400 as the only mode.

### Edge Cap Biases Cluster Visibility

The current cap keeps the top global edges by confidence and drops nodes not connected by retained edges. This favors dense/high-confidence areas and can shrink small or lower-confidence clusters disproportionately. The production 13-gene cluster 10 became 7 network nodes before category filtering.

### Clustering May Not Use Edge Confidence Explicitly

The Leiden call does not pass explicit weights. If the graph does not expose `combined_score` as `weight`, a medium-confidence edge and a high-confidence edge have the same community-detection effect. That reduces biological specificity and stability.

### Subcluster ID Parsing Is Ambiguous

Backend subcluster mode creates IDs such as `parent.subcluster`. The frontend parses a cluster string by splitting on `.` and using the parent number for grouping. That can be reasonable for parent grouping, but it is risky for subcluster legend, filtering, click handling, and table sync unless explicitly tested.

## Recommendations

### P0: Fast, Low-Risk Wins

1. Move the normal browser render path to Cytoscape `preset`, but preserve the current fCoSE representation.
   - Short proof: add `position: { x: node.x, y: node.y }` in `useNetworkData.ts` and use `preset` only as an opt-in/fallback test path for current igraph coordinates.
   - Production default: serve fCoSE-derived display coordinates from a cached API/worker artifact and render those with `preset`.
   - Keep client fCoSE as an explicit fallback when precomputed positions are missing or incomplete.
   - Expected impact: removes the 3-5s browser long task while keeping the current compound fCoSE visual grammar.

2. Use the typed analysis API client in `useNetworkData.ts`.
   - It currently uses raw axios, which violates the repository frontend boundary.
   - Switch to `getNetworkEdges()` from `app/src/api/analysis.ts`.

3. Separate true cluster size from visible network count.
   - Add API metadata per cluster: true cluster size, retained node count, visible count if possible.
   - UI labels should say "visible genes" when filters or edge caps are active.

4. Stop public page loads from triggering LLM batch generation.
   - Remove or gate the cache-hit `trigger_llm_batch_generation()` path in `jobs_endpoints.R`.
   - Trigger LLM summaries from admin jobs, scheduled prewarm, or "new hash detected" workflows.

5. Add regression tests for the `min_size >= 10` invariant.
   - Test main clusters and nested subclusters.
   - Test that fixtures named as functional clusters do not contain `cluster_size < 10` unless explicitly labeled as visual/mock data.

### P1: API And Data Model Improvements

1. Introduce an analysis snapshot/fingerprint.
   - Include algorithm version, STRING version, score threshold, min size, gene-set hash, STRING mapping hash, category/status hash, and layout version.
   - Pass the fingerprint into memoised functions so cache keys are data-aware.

2. Materialize derived network artifacts.
   - Store cluster memberships, enrichment summaries, network nodes, edges, and layout positions as versioned derived analysis artifacts.
   - Public API should read the current artifact, not compute it on demand.
   - Recompute via admin/worker when curation data or mappings change.

3. Split the initial table payload.
   - Initial load should return cluster summaries and first-page enrichment only.
   - Load full enrichment and identifiers for a selected cluster on demand.
   - Avoid returning all nested subclusters and all enrichment rows through job status.

4. Make edge capping cluster-aware.
   - Option A: keep all nodes and cap only edges, allowing isolated retained-cluster nodes.
   - Option B: allocate an edge budget per cluster plus a global high-confidence budget.
   - Option C: show a cluster centroid graph first, with on-demand cluster expansion.
   - In all cases, expose retained counts.

5. Propagate `min_confidence` into clustering or remove the implication.
   - If the request says `min_confidence = 700`, either cluster and edges should both use that threshold, or metadata should clearly state that cluster assignments are from the default 400-threshold cluster snapshot.

6. Use edge weights explicitly.
   - Confirm the STRING graph edge attribute names.
   - Pass `weights = combined_score / 1000` or a transformed confidence weight to Leiden and Walktrap where supported.
   - Record the weight source in metadata.

### P2: Data-Science Research Ideas

1. Run a stability grid.
   - Vary score thresholds: 400, 700, 900.
   - Vary Leiden resolution around 0.5, 0.75, 1.0, 1.25, 1.5.
   - Vary seeds or initial memberships.
   - Compare partitions using adjusted Rand index or normalized mutual information.
   - Track cluster-level Jaccard overlap across snapshots.

2. Consider a two-tier public network.
   - Default: high-confidence, stable core network.
   - Expand: medium-confidence exploratory network.
   - This aligns with STRING's confidence-score model and reduces noise in the default view.

3. Evaluate enrichment-first cluster naming.
   - Use stable hash IDs for internal identity.
   - Show human names from top enriched terms or curated labels.
   - Avoid relying on cluster numbers across releases.

4. Audit STRING ID collapses.
   - Report multiple HGNC IDs mapping to the same STRING protein.
   - Decide whether collapsed paralog/histone families need grouped node labels or explicit duplicate metadata.

5. Add release notes for analysis snapshots.
   - When cluster count or composition changes materially, publish a small provenance block: input gene count, STRING version, threshold, algorithm, min size, snapshot timestamp, and major cluster changes.

## Implementation Sketch

Recommended first implementation tranche:

1. Add position support to Cytoscape elements and make `useCytoscape.updateElements()` choose `preset` only when all gene nodes have finite positions.
2. Add a unit test asserting nodes with `x/y` get Cytoscape `position`.
3. Add a unit test asserting `useCytoscape.updateElements()` uses `preset` for fully positioned elements and fCoSE for missing/incomplete positions.
4. Add a worker/API proof-of-concept that computes fCoSE positions headlessly from the same Cytoscape element graph and returns `metadata.layout_engine = "cytoscape-fcose"`.
5. Update UI text to distinguish "visible genes" from true cluster size.
6. Remove LLM batch triggering from the public cache-hit path or guard it behind an admin/prewarm flag.
7. Add backend tests for no returned functional cluster or subcluster below `min_size`.

Second tranche:

1. Add data fingerprints to cache keys.
2. Add a content-addressed `gene_network_cytoscape_layout` artifact keyed by exact displayed elements and fCoSE option/version hashes.
3. Add minimal/paginated cluster summary endpoint for initial page load.
4. Add worker/admin prewarm for `functional_clustering`, `network_edges`, and fCoSE display layouts.
5. Store analysis snapshot metadata and expose it in the UI.

Third tranche:

1. Run the stability grid.
2. Decide default threshold and resolution from evidence.
3. Consider materialized analysis artifacts.
