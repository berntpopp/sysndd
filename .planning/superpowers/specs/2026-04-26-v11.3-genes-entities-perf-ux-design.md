# v11.3 — Genes/Entities Detail-Page Performance & UX Overhaul

**Date:** 2026-04-26
**Owner:** frontend
**Status:** spec — pending review (rev 3 after second-round review)

## 1. Problem

`/Genes/:symbol` (`app/src/views/pages/GeneView.vue`, 335 LoC) and `/Entities/:entity_id` (`app/src/views/pages/EntityView.vue`, 419 LoC) both fan out 8–10 API requests on mount and gate the entire page on a single `<BSpinner v-if="loading">`. The result: the user's curated SysNDD data (Associated Entities) paints **after** the slowest external lookup chain, the page feels sluggish, the layout wastes ~50 % of horizontal real estate at desktop widths, and several cards render an empty "no data" state instead of collapsing.

This spec covers the design contract for v11.3. Implementation plan(s) follow in `.planning/superpowers/plans/`.

### 2.3 Code-level findings

1. `<div v-if="loading">spinner</div>` gates the whole page until both gene calls return (`GeneView.vue:4`, `EntityView.vue:3`).
2. Two gene lookups (`?input_type=hgnc` and `?input_type=symbol`) fire concurrently as a fallback even when the URL parameter is the canonical symbol (`GeneView.vue:266`). The second call is wasted on the common path.
3. `<TablesEntities>` only needs the URL `:symbol` (or `:hgnc_id` — see §4.2.1), but is mounted only after the gene API resolves because `:filter-input` is computed from `geneData.value[0].symbol` (`GeneView.vue:172`).
4. **`TablesEntities.loadData()` debounces 50 ms before calling `doLoadData()`** (`TablesEntities.vue:617-620`). Any gate on "entity request starts after nav" must accommodate this 50 ms floor (or W2 must scope a small surgical edit — see §4.2.3).
5. External composables fire after `nextTick`, then issue ad-hoc `axios.get` per source with no shared cache; back-nav from a sibling gene re-fetches everything.
6. `EntityView.loadEntityInfo()` (`EntityView.vue:381`) `await`s the five sub-resources **sequentially** (status → review → publications → phenotypes → variation). Switching to parallel hooks is its own significant win.
7. Three nested fluid containers per section (`container-fluid > BContainer fluid > BRow > BCol`) accumulate ~30–40 px of stacked padding on every block, contributing to the sparse feel.
8. Sort-arrow visual noise at idle: `app/src/assets/scss/components/_tables.scss:168` already centralizes the both-arrows-stacked rule when `aria-sort='none'`. Fixing it is a small SCSS edit, not a new behavior.
9. Empty `Gene Constraint (gnomAD)` card always renders, even when its content is "No constraint data available for this gene".
10. **Already correct (do not regress):** `app/src/components/small/GenericTable.vue:8` already sets `stacked="md"` so `TablesEntities` (and every `GenericTable` consumer) collapses to per-row labelled stacks below `md`. Mobile table stacking is **not** an open issue.

## 4. Solution architecture

### 4.1 Data layer (no new runtime dep)

#### 4.1.1 `useResource<T>` contract

```ts
// app/src/composables/useResource.ts
import type { ComputedRef, Ref } from 'vue';

export type ResourceKey = string;
export type ResourceFetcher<T> = (signal: AbortSignal) => Promise<T>;

export interface ResourceState<T> {
  data: Ref<T | null>;
  error: Ref<Error | null>;
  loading: Ref<boolean>;        // true while a fetch is in-flight for this key
  isStale: Ref<boolean>;        // true when data is from cache and a revalidation is in-flight
  refresh: () => Promise<void>; // force-refetch ignoring TTL
  abort: () => void;            // unsubscribe this consumer; does NOT cancel the underlying fetch unless this is the last subscriber
}

export interface ResourceOptions {
  ttlMs?: number;                 // default 60_000 for our endpoints, 300_000 for externals
  staleWhileRevalidate?: boolean; // default true
}

export function useResource<T>(
  key: ResourceKey | ComputedRef<ResourceKey | null>,
  fetcher: ResourceFetcher<T>,
  opts?: ResourceOptions,
): ResourceState<T>;
```

#### 4.1.2 Multi-subscriber semantics

Two or more consumers that mount with the same `key` share one in-flight request via the cacheStore's pending-promise map. Concretely:

- The cacheStore tracks per-key: `{ value, fetchedAt, ttlMs, pending: Promise<T> | null, abortController: AbortController | null, refCount: number }`.
- On `useResource` mount: increment `refCount`. If `pending` exists, subscribe to it. Else if cache is fresh (within TTL), set `data` from cache and skip fetch. Else create a new `AbortController`, call `fetcher(controller.signal)`, store the promise in `pending`, and subscribe.
- On consumer unmount or key change: decrement `refCount`. If `refCount === 0` and a fetch is `pending`, call `abortController.abort()`. The fetcher receives the signal and is expected to honour it (axios accepts `AbortSignal` via `{ signal }` since v0.22).
- **Stale-result rule:** if a consumer's key changes mid-flight, the consumer marks its own subscription as cancelled. Even if the fetch eventually resolves (because another subscriber held it open), the cancelled consumer's `data` ref is **not** updated. The cacheStore still writes the resolved value into the cache map, so a later mount with that key sees fresh data.
- `refresh()` clears the cache entry's `fetchedAt` and triggers a new fetch on next tick; subscribers see `isStale = true` until the new value lands.
- `abort()` is the unsubscribe call; it never directly cancels the fetcher unless this consumer was the last subscriber.

#### 4.1.3 Pinia cacheStore

`app/src/stores/cacheStore.ts` — Pinia store wrapping a Map keyed by stable URL key with `{ value, fetchedAt, ttlMs, pending, abortController, refCount }`. LRU bounded at 64 entries (eviction skips entries with `refCount > 0`). TTL defaults: 60 s for "our" endpoints (gene record, entity sub-resources), 5 min for external endpoints (ClinVar, AlphaFold, MGI, RGD, UniProt).

#### 4.1.4 Per-source hooks

- `useGeneRecord(symbolOrHgncId)` — internally selects `?input_type=hgnc` for `/^HGNC:?\d+$/i.test(input)`, else `?input_type=symbol`. Single discriminated call.
- `useGeneClinVar(symbol)`, `useGeneAlphaFold(symbol)`, `useGeneUniProt(symbol)`, `useGeneMGI(symbol)`, `useGeneRGD(symbol)`.
- `useEntityRecord(entityId)`, `useEntityStatus(entityId)`, `useEntityReview(entityId)`, `useEntityPublications(entityId)`, `useEntityPhenotypes(entityId)`, `useEntityVariation(entityId)`.

The existing `useGeneExternalData` and `useModelOrganismData` are split into the per-source gene-* hooks (one source = one hook). The composables are removed once their last call site is migrated in W2.

**`TablesEntities` is *not* refactored to a hook.** It is a heavyweight Options-API component with internal pagination/filter/sort/URL-sync state; it owns its own `loadData()` that calls `listEntities()` (`TablesEntities.vue:510, :612`). v11.3 keeps that ownership and instead gives the page a way to mount it earlier. There is no `useGeneEntities` hook in this plan.

### 4.2 Hydration order

#### 4.2.1 Route-param shape and entity filtering

`/Genes/:symbol` accepts both an HGNC id (`/Genes/HGNC:4586`) and a symbol (`/Genes/GRIN2B`). The entity API supports both filter shapes — `equals(symbol,GRIN2B)` and `equals(hgnc_id,HGNC:4586)`. (Verified: a curl against `/api/entity/?filter=equals(hgnc_id,HGNC:4586)` returns the same 2 GRIN2B entities as the symbol filter.)

The page derives the filter directly from the URL parameter shape, with no dependency on the gene record:

```ts
// in GeneView.vue
const routeParam = computed(() => route.params.symbol as string);
const entityFilter = computed(() =>
  /^HGNC:?\d+$/i.test(routeParam.value)
    ? `equals(hgnc_id,${routeParam.value})`
    : `equals(symbol,${routeParam.value})`
);
```

`<TablesEntities :filter-input="entityFilter" :disable-url-sync="true">` therefore mounts and fires its `loadData()` on the same tick as the page mounts — gating on the gene record is removed. (No URL canonicalization needed; the entity API tolerates both shapes.)

#### 4.2.2 Mount order

1. **Entities first.** `<TablesEntities>` mounts immediately on page mount with the URL-derived filter; its `loadData()` is the first network request. No page-level loading gate.
2. **Gene header in parallel.** `useGeneRecord(routeParam)` fires on the same setup tick. The header card renders a skeleton until it resolves; cold target: skeleton ≤ 100 ms; resolved header within backend floor (gene API is ~50–100 ms warm).
3. **External hooks in parallel.** All five external hooks fire on the same setup tick, each owning its card's loading state. No `nextTick` deferral.
4. **Cancellation.** When the route param changes mid-flight (back/forward between genes), in-flight requests for the old key are aborted via the `useResource` `AbortController` (per §4.1.2 semantics).

#### 4.2.3 TablesEntities initial-load debounce

`TablesEntities.loadData()` (`TablesEntities.vue:617-620`) wraps `doLoadData()` in a 50 ms `setTimeout`. This protects against duplicate calls from chained reactivity — sound on its own merits — but it also means the entity request cannot start sooner than ~50 ms after `mounted()` runs, even with the gating gap removed.

W2 has two options to keep §8's "request starts ≤ 100 ms after nav" gate honest:

- **(A) Accept the 50 ms floor and target ≤ 100 ms after nav.** This is the default; no `TablesEntities` edit needed.
- **(B) Add a one-line `loadDataImmediate()` initial-load path** in `TablesEntities` that calls `doLoadData()` directly, bypassing only the very first `setTimeout`. Subsequent reactivity-triggered calls keep the debounce. This is a surgical edit, not a refactor of internals; §5 is updated to allow it explicitly.

W2 picks (A) unless the bench in W4 fails the gate, in which case W2 lands the (B) edit before closeout. The decision is recorded in W2's plan.

### 4.3 Skeletons + layout

- `app/src/components/ui/SectionCard.vue` — wrapper taking `:loading`, `:empty`, `:error`, slots for header/body/actions. Skeleton stripes while loading; renders nothing when `empty && !loading` (today's "No constraint data available" empty card disappears entirely). Stable height during skeleton phase to avoid CLS.
- Header band: gene badge + name + chromosome location + resources + identifiers in one wrapping flex row at ≥ xl breakpoint; stacks at smaller breakpoints. Resources and identifiers share the same badge component for visual consistency. Gene name becomes the largest text element on the page (lifts hierarchy score).
- Below header: Associated Entities (full width, unchanged size — it is the data-dense block).
- Below entities: 3-up `<BRow>` at `lg+` for `Constraint | ClinVar | ModelOrgs` cards. At `<lg` it stacks. Each card hides itself when its hook resolves to "no data".
- `GenomicVisualizationTabs`: lazy-mount inactive tab panels (today they all mount on load). `<KeepAlive>` so switching tabs is instant after first activation.
- Remove duplicate fluid wrappers: replace `container-fluid > BContainer fluid > BRow > BCol` triple-nesting with a single `BContainer fluid > BRow > BCol` per section.
- **Sort-arrow noise (existing fix):** edit `_tables.scss:168` so the `aria-sort='none'` state shows both arrows only on `:hover`/`:focus-visible`; idle renders neither (or the same neutral marker as before, faintly). No change to `BTable` props or `GenericTable.vue`.
- Footer: collapse the logo strip to a smaller footer row to recover ~40 px of below-the-fold real estate.

### 4.4 EntityView gets the same treatment

- The page mounts each section as its own `<SectionCard>` immediately on page mount; no page-level loading gate.
- Header card uses the same skeleton + `<SectionCard>` pattern with `useEntityRecord(entityId)`.
- `loadEntityInfo` is removed. Each subsection is owned by its own hook, all firing on the same setup tick:
  - `useEntityStatus(entityId)` → status badge / timeline card
  - `useEntityReview(entityId)` → review/synopsis card
  - `useEntityPublications(entityId)` → publications card (filters `publication_type === 'additional_references'`)
  - useEntityPublications result is reused for the gene-reviews card (filter `publication_type === 'gene_review'`); no second request
  - `useEntityPhenotypes(entityId)` → phenotypes card
  - `useEntityVariation(entityId)` → variation-ontology card
- The linked-gene block (showing the entity's HGNC gene) reuses `useGeneRecord(entity.hgnc_id)` once the entity record resolves.
- This explicitly fixes the existing sequential-await bug in `EntityView.loadEntityInfo` (Section 2.3 finding 6).

## 9. Verification harness

`app/tests/perf/genes-entities.bench.spec.ts` is a Playwright spec that:

1. Reads a fixture list of two genes (GRIN2B, MECP2) and two entities (`/Entities/304`, `/Entities/400`).
2. Calls `make cache-clear` via `node:child_process` (skip on CI; local-only) before the cold pass.
3. **Registers `PerformanceObserver`s before navigation** via `page.addInitScript`. Each observer is constructed with `{ buffered: true }` so any entries from before connection are flushed:
   ```ts
   await page.addInitScript(() => {
     (window as any).__perf = { lcp: 0, cls: 0, layoutShifts: [] };
     new PerformanceObserver((list) => {
       for (const entry of list.getEntries()) {
         if (entry.entryType === 'largest-contentful-paint') {
           (window as any).__perf.lcp = (entry as any).startTime;
         } else if (entry.entryType === 'layout-shift') {
           const e = entry as any;
           if (!e.hadRecentInput) {
             (window as any).__perf.cls += e.value;
             (window as any).__perf.layoutShifts.push({ t: e.startTime, v: e.value });
           }
         }
       }
     }).observe({ type: 'largest-contentful-paint', buffered: true });
     // separate observer for layout-shift to keep the buffered: true contract clean
     new PerformanceObserver((list) => {
       for (const entry of list.getEntries()) {
         const e = entry as any;
         if (!e.hadRecentInput) (window as any).__perf.cls += e.value;
       }
     }).observe({ type: 'layout-shift', buffered: true });
   });
   ```
4. For each probe, hard-navigates to the page; reads `performance.getEntriesByType('resource')` to capture entity-request start/end times; polls the DOM for skeleton frames present, first entity row, all spinners gone; reads `window.__perf` for LCP/CLS at "all settled".
5. Runs `@axe-core/playwright` against the loaded page; fails on new violations vs. baseline.
6. Writes timings to `.planning/perf/after-{date}.json` and a side-by-side diff vs. the baseline JSON in this PR.

The harness uses `PerformanceObserver` only — **`web-vitals` is not added** (per §5). INP is therefore not gated by the harness; LCP and CLS are.

The harness is **local-only** (per AGENTS.md, Playwright in this repo is local-only — there is no Playwright CI workflow). The acceptance numbers are committed in the perf JSON; PR review checks the diff.

## 11. Risks and mitigations

| Risk | Mitigation |
|------|------------|
| `useResource` SWR semantics drift from existing axios behaviour (interceptors, base URL, auth) | Build on top of the existing `apiClient` (`@/api/client`); inherit interceptors. Fetcher receives `AbortSignal` per §4.1.1; axios honours it via `{ signal }`. |
