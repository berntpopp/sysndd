# SEO Prerender Improvement Design

Date: 2026-05-14

## Goal

Improve SysNDD's public Vue SEO from a minimal gene/entity prerender baseline to a robust static-rendered public discovery layer for biomedical users and crawlers, without changing API/data/auth behavior or introducing a server-side rendering runtime.

## Current State

SysNDD already uses build-time prerendering. The frontend production build creates `app/dist/index.html`, then `app/scripts/generate-seo-pages.mjs` fetches compact SEO payloads and writes route-specific HTML for gene and entity detail routes. The generator also writes `sitemap.xml`, `sitemap-static.xml`, `sitemap-genes.xml`, and `sitemap-entities.xml`. `app/public/robots.txt` points crawlers to the sitemap.

The API supports:

- `GET /api/seo/routes`
- `GET /api/seo/gene/:symbol`
- `GET /api/seo/entity/:entity_id`
- `GET /api/seo/static`

The verified route scope is narrower than the public route map. API-backed generation prerenders `/Genes/:symbol`, `/Entities/:entity_id`, and `/`. It does not currently generate rich static shells for public collection, analysis, help, API, ontology, or search-intent pages. `svc_seo_static()` returns `/`, `/Genes`, and `/Entities`, but the generator only consumes `source.routes.static` from `/seo/routes`, where the current static list contains only `/`.

## External Guidance Applied

Google's JavaScript SEO guidance says app-shell pages require rendering before content is visible to crawlers, and that server-side or prerendering is beneficial because it is faster for users and crawlers and not every bot runs JavaScript. Google also says dynamic rendering is a workaround, not a long-term solution, and recommends server-side rendering, static rendering, or hydration instead.

Google structured data guidance requires structured data to be placed on the page it describes, use the most specific applicable Schema.org types and properties, and match visible page content. Dataset guidance recommends canonical dataset landing pages, sitemaps, and `sameAs` for duplicate dataset descriptions. Google snippet guidance supports programmatic descriptions for large database-driven sites when they are human-readable, diverse, and page-specific.

Unhead's `useSeoMeta()` is already available through the stack and is appropriate for client-side head consistency, but the SEO-critical source HTML must still be generated by the prerender pipeline.

PubTator evidence from CHD8 publications shows that biomedical discovery pages should expose gene symbols, diseases, phenotypes, species/model-organism context where relevant, and PMIDs as first-class facts. The current SysNDD payload already has much of this, but the generated HTML and JSON-LD underuse it.

## Design Principles

1. Static rendering first for public discovery content.
   The initial HTML for indexable public routes must contain route-specific title, description, canonical URL, visible facts, internal links, and JSON-LD before Vue runs.

2. No bot-only content.
   The prerender shell may be lighter than hydrated Vue, but its facts must match public visible data and must not invent content only for crawlers.

3. Biomedical entities over keywords.
   Pages should expose stable identifiers and relationships: HGNC, Ensembl, Entrez, OMIM, disease ontology, HPO, variation ontology, inheritance, classification, NDD status, PMIDs, and last modified dates.

4. Canonical URL discipline.
   Canonical links must be present in source HTML and match sitemap URLs. Query variants and duplicate access paths should converge on canonical paths.

5. DRY SEO builders.
   Metadata and HTML builders should be shared between TypeScript unit tests and the Node generator, or the generator should call one compiled source of truth. Avoid duplicated builder logic.

6. Verification must test generated HTML, not just builder functions.
   The SEO gate must inspect the actual files under `dist`, including route coverage, sitemap consistency, JSON-LD parseability, robots, and private-route exclusion.

## Target Route Coverage

### Tier 1 - Critical long-tail pages

- `/Genes/:symbol`
- `/Entities/:entity_id`

These remain the highest-value pages. They should be enriched first because they map directly to high-intent biomedical search queries.

### Tier 2 - Public collection and dataset landing pages

- `/`
- `/Genes`
- `/Entities`
- `/Phenotypes`
- `/API`
- `/Documentation`
- `/About`

These routes should have static shells with clear H1s, concise summaries, dataset context, and internal links to representative detail pages. `/Genes`, `/Entities`, and `/Phenotypes` should be treated as dataset collection pages, not empty SPA tables.

### Tier 3 - Public analysis and discovery pages

- `/Analysis`
- `/EntriesOverTime`
- `/CurationComparisons`
- `/PhenotypeCorrelations`
- `/PhenotypeFunctionalCorrelation`
- `/VariantCorrelations`
- `/PublicationsNDD`
- `/PubtatorNDD`
- `/GeneNetworks`

These should get route-specific metadata and concise crawlable descriptions of what the analysis shows. They do not need to prerender full charts.

### Tier 4 - Candidate future pages

- `/Ontology/:disease_term`
- selected `/Search/:search_term` pages only if there is a curated whitelist of stable high-value terms
- disease or phenotype landing pages derived from high-frequency public data

These should not be generated until canonicalization and content quality rules are clear.

Private/authenticated routes remain excluded and should not appear in sitemaps.

## Content Model

### Gene page

The generated source HTML should include:

- H1 with symbol and gene name.
- Summary paragraph: number of curated entities, leading diseases, inheritance modes, and classification spread.
- Definition list with identifiers: HGNC, Ensembl, Entrez, OMIM.
- Associated diseases with links to entity pages when available.
- Top phenotypes if available through public payloads.
- PMIDs as PubMed links, not just raw text.
- Internal links to related entity pages.
- Last modified date.

JSON-LD should include a WebPage node with `about` Gene-like data, `identifier`, `citation`, `isPartOf` WebSite/DataCatalog, and breadcrumb data. Use Schema.org terms conservatively and only when the visible page supports them.

### Entity page

The generated source HTML should include:

- H1 with gene and disease.
- Summary paragraph: entity identifier, disease, inheritance, classification, NDD status.
- Compact facts for disease ontology, inheritance, classification, NDD status.
- Clinical synopsis if public and approved.
- HPO and variation terms.
- PMIDs as PubMed links.
- Internal links to the gene page and related entity pages.
- Last modified date.

JSON-LD should include MedicalWebPage plus a linked `about` graph for gene, disease/condition, phenotype terms, citations, and breadcrumb.

### Dataset and collection pages

The generated source HTML should include:

- H1 and summary of what the collection contains.
- Current counts from API-backed SEO stats.
- Links to canonical collection routes and representative detail pages.
- Dataset/DataCatalog JSON-LD for the public SysNDD dataset, including publisher, license/citation URL if available in existing app content, and distribution links if already public.

Do not expose private curation statistics or admin-only data.

## Architecture

Keep the build-time prerender approach. Do not move to Nuxt or a Node SSR server in this iteration.

Refactor SEO generation into focused modules:

- Route payload services in `api/services/seo-service.R`
- Endpoint annotations in `api/endpoints/seo_endpoints.R`
- Shared frontend SEO types and builders in `app/src/seo/`
- Generator entrypoint in `app/scripts/generate-seo-pages.mjs`
- Verification scripts in `app/scripts/`

The preferred implementation is to make the generator import or consume one shared JavaScript/TypeScript SEO builder output rather than duplicating builder logic in `generate-seo-pages.mjs`.

## Verification Requirements

Run:

- `make verify-seo-app`
- `make lint-app`
- `cd app && npm run type-check`
- `cd app && npm run type-check:strict`
- `cd app && npm run test:unit`
- API unit test for `test-seo-endpoints.R`

Enhance `make verify-seo-app` or its underlying scripts to validate:

- generated files exist for fixture static, gene, and entity routes
- canonical URLs match sitemap URLs
- sitemap excludes authenticated/private routes
- `robots.txt` exists and points to `sitemap.xml`
- JSON-LD parses as JSON
- source HTML contains non-empty H1, description, canonical, and visible route-specific facts
- no generated page keeps the generic SPA title
- generated route count matches fixture route count

Manual external checks after deployment:

- Google Rich Results Test for representative gene/entity/dataset pages
- URL Inspection for representative pages in Search Console
- sitemap submission/refresh

## Non-Goals

- No ranking guarantees.
- No clinical decision support content.
- No private/admin/curation route indexing.
- No runtime SSR migration.
- No bot-specific alternate content.
- No broad search-result page generation without a curated whitelist.

## Risks

- Duplicate builder logic can drift from tested TypeScript code.
- Sitemap and canonical mismatches can create duplicate URL signals.
- Thin collection-page shells can look like doorway pages if they do not expose real public value.
- Large production route counts may eventually need sitemap chunking beyond the current three-file layout.
- Build-time generation can silently be skipped if deployment does not set the SEO generation path.

## Success Criteria

- Public source HTML for critical pages is useful without JavaScript.
- `/Genes`, `/Entities`, `/Phenotypes`, `/API`, `/Documentation`, and `/About` have route-specific source HTML.
- Gene and entity pages expose PMIDs, identifiers, and biomedical relationships more richly.
- The generator has a single metadata/content builder source of truth or equivalent integration coverage that prevents drift.
- Verification catches missing routes, generic metadata, private routes in sitemaps, invalid JSON-LD, and robots/sitemap regressions.
