### W3 — Issue #29 entity-batch grouping fix

**Source file (primary):** `api/services/re-review-service.R`.

**Root cause:** `batch_preview` and `batch_create` use `SELECT DISTINCT e.entity_id ... ORDER BY r.review_date ASC LIMIT ?`. A gene with multiple `entity_id`s (one per disease association — e.g., GNAO1, GRIN1, IFT172 from the issue report) carries independent `review_date`s on each entity. When `LIMIT` cuts mid-gene, half its entities land in batch N and the rest fall to batch N+1. `DISTINCT` is row-level; nothing keeps a gene atomic.

**Fix algorithm — gene-first selection then expand:**

1. With the same WHERE clause `criteria` produces, first SELECT distinct `hgnc_id`s that have at least one matching entity, ordered by oldest matching `review_date` for that gene. Cap the gene-id list at `batch_size` (worst case 1 entity per gene).
2. SELECT all entities matching the criteria where `hgnc_id IN (the gene list)`, ordered by `hgnc_id, review_date`.
3. Trim with a soft LIMIT — include all entities for a partially-included gene; stop adding new genes once cumulative entity count ≥ `batch_size`. The actual returned count can slightly exceed `batch_size` for the boundary gene; that is the price of atomicity.

**Why this over a SQL window-function approach:** R-side soft-LIMIT logic stays out of MySQL — easier to unit-test in `testthat`, easier to extend if the cap rules change.

**Preview-UI hint:**

- `batch_preview` response gains three fields: `boundary_gene` (the gene where soft-LIMIT engaged, or `null` if the batch fits cleanly), `gene_count`, `entity_count`.
- `app/src/views/curate/ManageReReview.vue` already calls preview before create. On a non-null `boundary_gene`, render a `BAlert` variant=`warning` near the create button: "Batch is gene-atomic. Currently the batch will hold M entities to keep gene K together (requested N). The next batch will pick up the rest of gene K's matches if any remain. Tighten criteria to avoid the overflow."

**W3 task breakdown** (final detail in W3's plan file):

- W3.1 — Worktree + write a failing R test reproducing the split-gene scenario (seed two genes with 3 entities each, set criteria that match all 6, request `batch_size = 4`; assert the result is 3 entities for one gene OR 6 entities for both, never 4 split across genes).
- W3.2 — Implement the two-step gene-first algorithm in `batch_preview`. Verify whether `batch_create` shares `batch_preview`'s selection logic or has its own; patch whichever drives the actual SELECT. Mirror the soft-LIMIT in both.
- W3.3 — Extend `batch_preview` response with `boundary_gene` / `gene_count` / `entity_count`. Add or update the testthat spec covering happy-path response shape.
- W3.4 — Frontend: extend `ManageReReview.vue` preview consumer to render the warning `BAlert` when `boundary_gene` is non-null. Add a Vitest spec asserting the alert renders only when the field is populated.
- W3.5 — `make lint-api`, `cd api && Rscript -e "testthat::test_file('tests/testthat/test-re-review-service.R')"`, `cd app && npx vitest run src/views/curate/ManageReReview.spec.ts`, `make pre-commit`. Push, open PR, gate on CI.

**Risks & mitigations:**

- **Existing in-flight batches** are not corrected. A batch already created with split-gene state stays as-is. Reviewers handle this manually if/when they encounter it. No DB migration. (Per design choice — algorithm-fix-only scope.)
- **Soft-LIMIT can return entities > batch_size.** Search the codebase for `batch_size` and `batch_preview` callers to confirm none assume a strict LIMIT for sizing UI elements. `ManageReReview.vue` preview uses the response length, not the requested cap, for display — so safe. Document the behavior change in `batch_preview`'s docblock and add a one-line entry to `AGENTS.md` § "Stack-Specific Gotchas".
- **Performance.** The gene-list step is one extra query per preview. Both queries hit `ndd_entity_view` joined to `re_review_*`; the WHERE is the same; the index hits are the same. Acceptable.
