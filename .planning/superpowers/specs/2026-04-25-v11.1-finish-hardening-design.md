#### Fixture and data contract (added per spec review finding 3)

The Playwright suite runs against a real API + real DB. Determinism is the contract:

**Stack startup (CI):**

- A new make target `make playwright-stack` brings up the API + DB + frontend (built `dist/`, served by nginx — same image as production) on a deterministic compose project name. Defined in a new `docker-compose.playwright.yml` overriding ports/volumes off `docker-compose.yml`.
- The `playwright.yml` workflow runs `make playwright-stack`, waits for `/api/version` 200, runs Playwright, tears the stack down via `docker compose -p playwright down -v`.
- Local equivalent: developers run `make playwright-stack` then `cd app && npx playwright test`.

**Test users:**

A new SQL fixture `db/fixtures/playwright_users.sql` (sourced from `make playwright-stack`) provisions four deterministic accounts:

| Username | Role | Purpose |
|---|---|---|
| `pw_admin` | Administrator | admin.manage-user.spec.ts |
| `pw_curator` | Curator | entity.create / entity.modify |
| `pw_reviewer` | Reviewer | review.approve / status.approve |
| `pw_user` | Viewer | auth flows that don't require elevated role |

Passwords are stored hashed (bcrypt, same salt rounds as production). The plaintext is committed in `app/tests/e2e/fixtures/test-users.ts` because these accounts only exist in the Playwright stack and never reach production. The fixture ID/salt scheme guarantees the bcrypt hash in the SQL fixture is reproducible.

**Auth fixture (programmatic login):**

`app/tests/e2e/fixtures/auth.ts` exports `loggedInAs(role: 'admin' | 'curator' | 'reviewer' | 'user'): Promise<BrowserContext>` — calls `POST /api/auth/authenticate` directly to obtain a token, injects it into `localStorage` for the new context, and returns the context. This bypasses UI for non-auth tests, keeping the auth flows the only place that exercises the login form.

**Per-test data isolation:**

- Each test that creates entities, reviews, statuses, or users uses a unique suffix derived from `Date.now()` + `crypto.randomUUID()` (helper `uniqueName(prefix)` in `fixtures/data.ts`)
- No test deletes another test's data; cleanup is handled by stack tear-down between full runs
- Tests must be order-independent; Playwright's parallel execution (default workers) is enabled

**Reset between full runs:**

- CI: `docker compose -p playwright down -v` removes volumes; next run starts from a clean fixture
- Local: same teardown commanded by `make playwright-stack-down` (added)
- During a single run: no reset; tests rely on uniqueness

**Failure capture:**

- Playwright `trace: 'retain-on-failure'`, `video: 'retain-on-failure'`, `screenshot: 'only-on-failure'` for failure cases
- Successful-run screenshots (the artifact-only screenshots from §3 of the review's bucket C) are captured by explicit `await page.screenshot({ path: ... })` calls inside specs at meaningful checkpoints

#### Screenshot handling

- Output dir: `app/tests/e2e/screenshots/` — gitignored
- Naming: `<flow>__<step>__<viewport>.png`
- CI: Playwright job uploads `app/tests/e2e/screenshots/` and Playwright traces as workflow artifacts, retention 14 days
- No pixel-diff assertion; screenshots exist for human review of intentional UI changes

#### Exit

Green CI run on integration with both `playwright` (deep flows) and `playwright-smoke` jobs reported. `make playwright-stack` works locally on a clean dev box.

## 5. Cross-cutting contracts

### Typed-client conventions (W3 output, consumed by W4/W5/W6/W7)

Template (modeled on existing `app/src/api/external.ts`):

```ts
import type { AxiosRequestConfig } from 'axios';
import { apiClient } from './client';

export interface FooResponse { /* mirror R endpoint shape */ }

/**
 * GET /api/foo/<id>
 * Mirrors api/endpoints/foo_endpoints.R:NN
 * Returns FooResponse on 2xx; throws AxiosError on non-2xx.
 */
export async function getFoo(id: string, config?: AxiosRequestConfig): Promise<FooResponse> {
  const path = `/api/foo/${encodeURIComponent(id)}`;
  return apiClient.get<FooResponse>(path, config);
}
```

Rules:

- Helpers throw on non-2xx — no `{ ok, data, error }` wrappers
- 404-as-no-data is a *caller* concern, documented in JSDoc, never swallowed in the helper
- Response types mirror the R endpoint exactly; if the R serializer wraps scalars in 1-element arrays, the helper unwraps and the type is the unwrapped form
- Path params go through `encodeURIComponent`; query params pass via `config.params`, never string concatenation
- No retry/circuit-breaker logic — callers own that

### Error-handling pattern (W2 + all consumers)

- `apiClient` interceptor maps non-2xx → throws `AxiosError`
- 401: interceptor calls `useAuth().handle401()` then re-throws so callers can also branch
- Callers use `try/catch` and `isApiError(err)` (helper added in `app/src/api/client.ts` if not already present)
- No `.catch(() => null)` swallow patterns introduced in this PR

### Migration pattern (W4/W5/W6/W7 mechanical guidance)

For every `this.axios.<verb>(...)` site in an Options API view:

1. Identify the R endpoint from the URL string (e.g., `/api/genes/<id>` → `getGene` in `api/genes.ts`)
2. Add `import { getGene } from '@/api/genes'` to `<script>`
3. Replace `const response = await this.axios.get(apiUrl); const data = response.data;` with `const data = await getGene(id);`
4. Drop the `apiUrl` const if no longer used
5. Drop value-imports of `axios` if no longer used; type-only imports may remain
6. If file is in strict scope, run `npm run type-check:strict`

The pattern is mechanical. No "should we rewrite this view" judgment in W4/W5/W7 — that's reserved for follow-ups (or W6, for `Review.vue` only).

## 8. Risk register

| # | Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|---|
| R1 | Playwright in CI is slow / flaky on first runner setup | Medium | Medium | Wave 0 explicitly includes CI green as exit gate; if flaky, isolate with `test.fixme` and document |
| R2 | CSP tightening breaks third-party scripts (analytics, fonts, Plotly inline) we forgot about | Medium | High | W1's audit script catches build-time inline content; security-headers spec asserts response headers; manual click-through of charts/maps before merge |

| R4 | `Review.vue` decomposition (W6) introduces subtle reactivity bugs that pass Vitest but break in browser | Medium | High | Playwright `review.approve.spec.ts` is the explicit gate; if it goes red, decomposition reverts and stays migration-only |

| R6 | Worktree owners step on `app/src/api/*.ts` despite the rule | Low | Medium | Wave gating + owner-side `grep` checks before merge; rebase conflicts surface it loudly |
| R7 | `make ci-local` host-side R quality target fails on a dev box without `HOST_R_LD_LIBRARY_PATH` | Medium | Low | Already documented in `AGENTS.md`; surface in plan-doc setup notes |

## 9. Rollback strategy

| Failure mode | Recovery |
|---|---|
| One Wave 1a sub-branch flaky | Hold its merge; the others can still merge. The slow one rebases and re-tests. |
| Wave 1a integration merge breaks Playwright | `git revert -m 1 <merge-commit>` on integration, rebase the offending sub-branch with the regression test, re-merge |
| Wave 1b discovers W3 contract bug | Hotfix on `fh/typed-clients-fix` (one-off worktree off integration), merge before continuing 1b |
| Playwright baseline itself proves wrong/flaky in Wave 0 | Roll back Wave 0 commit chain on integration, redesign affected specs, re-land |
| Catastrophic — abandon the PR | Delete the integration branch and all `fh/*` worktrees; no master pollution. Playwright baseline can be salvaged into a separate PR |

Hard rules during rollback:

- Never `git reset --hard` the integration branch — only `git revert`
- Never force-push integration
- Worktree owners do not rebase across a revert without confirming with the integration owner first

## 11. External infrastructure

- Playwright needs Node + browsers in CI runner. `ubuntu-latest` is sufficient for Chromium.
- Playwright stack needs Docker. Workstream owners need Docker locally for full E2E verification, otherwise rely on `make pre-commit` + CI for E2E coverage.
- New compose file `docker-compose.playwright.yml` and SQL fixture `db/fixtures/playwright_users.sql` are CI-only artifacts; they MUST NOT be sourced by `make dev`, `make ci-local`, or any production migration path. CI greps production migration paths to enforce this (`grep -r 'playwright_users' db/migrations/` returns nothing).
- No new database migrations in this PR. The Playwright user fixture is a SQL seed, not a migration. If any workstream feels the need for a migration, escalate — it is a scope-creep signal.
