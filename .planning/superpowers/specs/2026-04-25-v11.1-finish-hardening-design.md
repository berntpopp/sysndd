## 5. Cross-cutting contracts

### Typed-client conventions (W3 output, consumed by W4/W5/W6)

Template (modeled on existing `app/src/api/external.ts`):

```ts
import type { AxiosRequestConfig } from 'axios';
import { apiClient } from './client';

export interface FooResponse { /* mirror R endpoint shape */ }

/**
 * GET /api/foo/<id>
 * Mirrors api/endpoints/foo_endpoints.R:NN
 * Returns FooResponse on 2xx; throws AxiosError on non-2xx.
 */
export async function getFoo(id: string, config?: AxiosRequestConfig): Promise<FooResponse> {
  const path = `/api/foo/${encodeURIComponent(id)}`;
  return apiClient.get<FooResponse>(path, config);
}
```

Rules:

- Helpers throw on non-2xx — no `{ ok, data, error }` wrappers
- 404-as-no-data is a *caller* concern, documented in JSDoc, never swallowed in the helper
- Response types mirror the R endpoint exactly; if the R serializer wraps scalars in 1-element arrays, the helper unwraps and the type is the unwrapped form
- Path params go through `encodeURIComponent`; query params pass via `config.params`, never string concatenation
- No retry/circuit-breaker logic — callers own that

### Error-handling pattern (W2 + all consumers)

- `apiClient` interceptor maps non-2xx → throws `AxiosError`
- 401: interceptor calls `useAuth().handle401()` then re-throws so callers can also branch
- Callers use `try/catch` and `isApiError(err)` (helper added in `app/src/api/client.ts` if not already present)
- No `.catch(() => null)` swallow patterns introduced in this PR

### Migration pattern (W4/W5 mechanical guidance)

For every `this.axios.<verb>(...)` site in an Options API view:

1. Identify the R endpoint from the URL string (e.g., `/api/genes/<id>` → `getGene` in `api/genes.ts`)
2. Add `import { getGene } from '@/api/genes'` to `<script>`
3. Replace `const response = await this.axios.get(apiUrl); const data = response.data;` with `const data = await getGene(id);`
4. Drop the `apiUrl` const if no longer used
5. Drop the `axios` import if it was the last user
6. If file is in strict scope, run `npm run type-check:strict`

The pattern is mechanical. No "should we rewrite this view" judgment in W4/W5 — that's reserved for follow-ups (or W6, for `Review.vue` only).

|---|---|---|---|

## 9. Rollback strategy

| Failure mode | Recovery |
|---|---|
| One Wave 1a sub-branch flaky | Hold its merge; the other three can still merge. The slow one rebases and re-tests. |
| Wave 1a integration merge breaks Playwright | `git revert -m 1 <merge-commit>` on integration, rebase the offending sub-branch with the regression test, re-merge |
| Wave 1b discovers W3 contract bug | Hotfix on `fh/typed-clients-fix` (one-off worktree off integration), merge before continuing 1b |
| Playwright baseline itself proves wrong/flaky in Wave 0 | Roll back Wave 0 commit chain on integration, redesign affected specs, re-land |
| Catastrophic — abandon the PR | Delete the integration branch and all `fh/*` worktrees; no master pollution. Playwright baseline can be salvaged into a separate PR |

Hard rules during rollback:

- Never `git reset --hard` the integration branch — only `git revert`
- Never force-push integration
- Worktree owners do not rebase across a revert without confirming with the integration owner first
