## Scope

This design covers six safe work areas:

1. Issue-driven documentation clarifications for #49 through #52.
2. Factual freshness updates for stale runtime, stack, route, and analysis descriptions.
3. A first documentation screenshot automation lane using Playwright against the Docker-backed SysNDD app/API/DB stack.
4. A gradual screenshot migration from legacy manual images to generated assets.
5. A restrained documentation design overhaul for the Quarto book.
6. Optional CI integration after the local screenshot workflow is stable.

The screenshot automation work must be a separate implementation phase from the content clarification work.

Existing documentation build artifacts and logs, such as generated `_book` or `_site` outputs and `SysNDD_documentation.log`, are intentionally not a first-phase acceptance gate for #49 through #52 or #140. They may be handled as low-risk artifact-hygiene cleanup during the content freshness or documentation design phases if verified against the repository ignore/build policy.

## Non-Goals

- Do not change SysNDD curation principles, thresholds, evidence categories, category meanings, or NDD scope.
- Do not introduce new curation categories, variation ontology terms, thresholds, or decision rules.
- Do not infer molecular mechanisms beyond source text in the variant ontology documentation.
- Do not migrate from Quarto to another documentation framework.
- Do not migrate the app to SSR or change public UI behavior as part of this documentation work.
- Do not treat generated documentation screenshots as Playwright visual-regression baselines or failure artifacts.
- Do not delete all legacy screenshots in the first automation pass.
- Do not add always-on screenshot generation to CI before local generation is proven stable.

## Content Design

### Issue #49: Browser Console Bug Reporting

Add user-facing guidance in `documentation/02-web-tool.qmd` near the existing "Reporting bugs, problems and making feature requests" section.

The section should stay practical and non-developer-facing. It should ask for a minimal problem report containing:

- Page URL or route.
- Date, time, and browser.
- Logged-in role if relevant.
- Exact steps to reproduce the problem.
- Expected result.
- Observed result.
- Visible error message, if present.
- Console errors or saved console log.
- Whether the problem repeats after reload.

Add a short Chromium browser console workflow:

- Open Developer Tools with `F12`, `Ctrl+Shift+J`, or the browser menu.
- Select the Console tab.
- Reproduce the issue.
- Save or copy console output.
- Send the report through the current support route.

Include a privacy warning: users should not send passwords, tokens, personal identifiers, unpublished patient-level details, or other sensitive material in console logs or screenshots.

### Issue #50: Finding A PMID For A GeneReviews Article

Add reviewer-facing guidance in `documentation/06-re-review-instructions.qmd` near the existing GeneReviews PMID instruction.

The content should explain:

- GeneReviews chapters are hosted in NCBI Bookshelf and may have PubMed records.
- Reviewers should use the chapter-specific PMID when one exists.
- Reviewers should not substitute the general GeneReviews collection PMID for a chapter-specific citation.
- If no chapter-specific PMID can be found, reviewers should note that in the review comment rather than forcing an unrelated PMID.

Provide two workflows:

- From GeneReviews or NCBI Bookshelf: open the chapter, check the citation or metadata area, and record the PubMed PMID when present.
- From PubMed: search the condition name plus `GeneReviews`, open the matching chapter record, and copy the PMID.

### Issue #51: Variant Ontology Curation Description

Add explanatory prose in `documentation/06-re-review-instructions.qmd`, with a cross-reference from `documentation/04-database-structure.qmd`.

The prose should describe variant ontology curation as source-based standardization of the molecular consequence or mechanism already supported by the cited literature. It should not create or rename terms.

Recommended guidance:

- Read the publication, GeneReviews chapter, OMIM entry, or other accepted source text.
- Identify the variant class or mechanism explicitly supported by the source.
- Select the closest existing SysNDD variation ontology term.
- Prefer a more specific existing term only when the source is explicit.
- Use broader wording or a curator comment when sources are heterogeneous or ambiguous.
- Do not infer a mechanism beyond the source text.
- Use the SysNDD API endpoint, VariO, or OLS as the authoritative term source.

Examples must be illustrative only:

- Recurrent missense variants with an explicitly reported gain-of-function mechanism can be captured with the matching existing mechanism or effect term if present.
- Truncating variants reported as causing loss of function can be captured with the matching existing loss-of-function or truncating concept if present.
- Mixed missense and truncating reports without a proven common mechanism should be described conservatively in synopsis or comments rather than forcing an over-specific ontology assignment.

### Issue #52: `n.a.` Category Explanation

Expand `documentation/05-curation-criteria.qmd` in the existing "Special case: non-NDD entities" section. The change must explain the existing meaning without changing the rule.

The text should define `n.a.` as a tag for a retained gene-disease-inheritance entity that is outside SysNDD's NDD classification scope, often because the same gene has other NDD-relevant entities and the non-NDD entity remains useful for gene context.

Clarify contrasts:

- Category 1, 2, and 3 ask how strong the evidence is for an NDD entity.
- `n.a.` says the entity itself is outside the NDD classification scope.
- "Limited" is weak but plausible NDD evidence.
- "Refuted" is evidence against retaining the proposed association.
- `n.a.` is not a lower evidence category.

Examples must remain explanatory applications of existing policy:

- A gene has one entity with early-onset developmental delay or ID and another well-established entity without cognitive impairment; the non-NDD entity may be retained for context and tagged `n.a.`.
- A disease is mainly adult-onset or organ-specific and the literature does not show ID or NDD in a significant fraction; if retained as a separate entity, it may be tagged `n.a.`.
- A disorder includes motor delay only, with no evidence for cognitive impairment or early neurodevelopmental disorder; the issue may be scope rather than weak NDD evidence.
- Do not use `n.a.` when there is weak but plausible NDD evidence; apply Category 2 or 3 according to existing criteria.
- Do not use `n.a.` when newer evidence contradicts the old association; consider "Refuted" according to existing criteria.

## Documentation Design

Keep the Quarto book. The design work should make the documentation feel consistent with SysNDD's current clinical, compact, table-first product direction.

Design principles:

- Quiet, readable, operational documentation.
- No marketing hero treatment.
- No decorative gradients or card-heavy landing redesign.
- Consistent image widths and captions.
- Captions should explain the task or state being shown, not restate the filename.
- Screenshots should usually show the product surface without browser chrome unless browser or OS UI is the subject.
- Generated callouts should be restrained and used only when they reduce explanatory text.

The design overhaul should be planned after content clarifications and initial screenshot automation so the visual work can use current images rather than re-styling stale documentation.

## Screenshot Automation Design

### VarLens Pattern To Adopt

VarLens provides a useful pattern:

- A named `docs:screenshots` command.
- A make target wrapping documentation screenshot generation.
- Stable screenshot slugs.
- Deterministic data setup.
- Playwright-driven navigation.
- DOM overlays or generated annotation metadata.
- Generated screenshots committed as documentation assets.
- Optional CI artifact handoff before documentation build.

### VarLens Pattern Not To Copy Directly

SysNDD must adapt the pattern because it is not an Electron/VitePress app.

Do not copy:

- Electron launch logic.
- VitePress public screenshot paths.
- CI-first generation.
- One large screenshot test file that mixes manifest, setup, overlay, and capture logic.

SysNDD needs a Docker-backed web stack: app, API, MySQL, Traefik, deterministic fixtures, and role-based authentication.

### SysNDD Screenshot Lane

Create a dedicated docs screenshot lane under `app/tests/docs-screenshots/`. It should run with one worker and should not be part of the default local-only E2E suite unless invoked directly.

Because the main Playwright config has `testDir: './tests'`, isolation must be explicit. The implementation plan should choose one of these safeguards:

- A dedicated config such as `app/playwright.docs-screenshots.config.ts`, used only by `npm run docs:screenshots`.
- A default-suite `testIgnore` or `testMatch` rule that cannot collect `tests/docs-screenshots/**`.
- A separate Playwright project that is excluded from default `npm run test:e2e` execution and included only by the docs screenshot command.

Do not rely only on directory placement for isolation.

Proposed structure:

- `manifest.ts`: stable screenshot definitions.
- `docs-screenshots.spec.ts`: Playwright runner that iterates the manifest.
- `overlays.ts`: DOM overlay helpers for boxes, dots, callouts, masks, and cleanup.
- `provenance.ts`: generated metadata writer.

The lane should use the existing Playwright stack:

- `make playwright-stack`
- `cd app && PLAYWRIGHT_BASE_URL=http://localhost npm run docs:screenshots`
- `make playwright-stack-down`

The first implementation may expose a root target such as `make docs-screenshots` once lifecycle behavior is clear. The plan should decide whether teardown is explicit or trap-based.

### Manifest Model

Each screenshot entry should include enough data to regenerate and audit the image:

```ts
type DocsScreenshotAction =
  | { type: 'click'; selector: string }
  | { type: 'fill'; selector: string; value: string; sensitive?: boolean }
  | { type: 'press'; key: string }
  | { type: 'hover'; selector: string }
  | { type: 'waitFor'; selector: string }
  | { type: 'callHelper'; name: string; args?: Record<string, unknown> }

type DocsScreenshot = {
  slug: string
  output: string
  docRefs: string[]
  route?: string
  url?: string
  baseURL?: 'app' | 'api' | string
  viewport: { width: number; height: number }
  authRole?: 'admin' | 'curator' | 'reviewer' | 'user'
  setup?: string
  actions?: DocsScreenshotAction[]
  waitFor?: string
  locator?: string
  fullPage?: boolean
  clip?: { x: number; y: number; width: number; height: number }
  maskSelectors?: string[]
  annotations?: Array<{
    selector: string
    mode: 'box' | 'dot' | 'callout'
    label?: string
    number?: number
  }>
}
```

`route` is for SPA-relative navigation. `url` or `baseURL` is for non-SPA surfaces such as the API/Swagger UI or any future documentation page capture. Interactive surfaces such as reviewer pages, modals, menus, and Swagger authorization states must be represented through named `setup` helpers and serializable `actions`, not through ad hoc logic hidden inside the runner.

The generated provenance manifest should include:

- Screenshot slug.
- Output path.
- Source route.
- Referencing documentation files or intended sections.
- Base target, when not the default app base URL.
- Viewport.
- Auth role.
- Setup helper and action sequence, with sensitive action values redacted.
- Wait condition.
- Capture mode.
- Annotation metadata.
- Git SHA if available.
- Capture timestamp.
- Browser name and version if practical.

### Output Policy

Generated documentation screenshots should live under `documentation/static/img/generated/`.

They are documentation assets, not Playwright artifacts:

- They should not be written to `app/tests/e2e/.playwright-output/`.
- They should not use `toHaveScreenshot()` visual-regression baselines.
- They should be referenced by Quarto when migrated.
- They may be committed when they are used by published docs.

Raw captures under `documentation/static/img/generated/raw/` are optional. The critical requirement is that generated final images and provenance metadata can be reproduced from the manifest.

### Stability Controls

The screenshot runner should:

- Use fixed viewports.
- Run with one worker.
- Use deterministic Playwright users and fixture data.
- Pre-acknowledge the disclaimer through existing fixture patterns.
- Use explicit waits for route readiness.
- Hide or mask volatile elements such as timestamps, usernames, counts, or environment-specific values when needed.
- Prefer role or accessible-name selectors where practical.
- Keep any state mutation minimal and resettable.
- Avoid screenshots that require private, unpublished, or sensitive data.

### First Capture Set

The first automation phase should prove breadth, not replace every image.

Recommended first set:

- Home page.
- Entities table.
- Genes table.
- Gene detail page.
- Entity detail page.
- Compare curations overview.
- NDD Publications analysis.
- Functional clusters analysis.
- Login page.
- Reviewer re-review page.
- Modify review modal.
- API Swagger authentication screen.

This covers public, analysis, auth, reviewer, modal, and API surfaces before scaling migration to all legacy images. The reviewer and modal shots require manifest actions or setup helpers; the Swagger shot requires a non-SPA URL or base override.

### Screenshot Verification

Add a lightweight verifier after the first generator exists.

The verifier should check:

- Every generated manifest entry has an output image.
- Every generated output image is represented in the generated provenance manifest.
- Quarto image references resolve.
- Orphaned legacy screenshots are reported.
- Stale manual screenshots are allowed while migration is incomplete.

The verifier can live under `app/scripts/` or `scripts/documentation/`. The later implementation plan should choose the location based on repository conventions.

### CI Integration

CI integration is optional and late.

Preferred sequence:

1. Local/manual generation.
2. Manual `workflow_dispatch` screenshot job.
3. Artifact upload/download before Quarto render.
4. Optional scheduled or PR-comment workflow after stability is proven.

Do not auto-commit generated screenshots from CI in the first CI phase.

## Risks And Mitigations

Risk: Curation prose accidentally changes policy.

Mitigation: Keep #51 and #52 explicitly explanatory. Review changed curation sections against the original thresholds and category definitions.

Risk: `n.a.` examples are read as new decision rules.

Mitigation: Frame examples as applications of existing scope distinctions. Avoid prescriptive language beyond the existing criteria.

Risk: Variant ontology prose suggests unsupported mechanistic inference.

Mitigation: Repeat that curators should use the source text and existing terms only.

Risk: Screenshot generation becomes flaky.

Mitigation: Use fixed viewports, one worker, deterministic fixtures, explicit waits, and masking for volatile data.

Risk: The first automation PR becomes too large.

Mitigation: Capture 8-12 representative screenshots first and keep legacy images until each section is migrated.

Risk: CI screenshot generation is too heavy for the Docker-backed stack.

Mitigation: Keep CI optional and manual until local generation is stable.

Risk: Documentation design work hides content regressions.

Mitigation: Do content and automation first, then design cleanup, with `quarto render documentation` and image-reference verification after each relevant phase.

## Acceptance Criteria

Phase 1 - issue-driven documentation clarification acceptance:

- #49 is addressed with a minimal problem report checklist and Chromium console-log workflow.
- #50 is addressed with GeneReviews PMID lookup guidance that distinguishes chapter-specific PMIDs from collection-level records.
- #51 is addressed with source-based variant ontology curation prose and examples, without adding or renaming ontology terms.
- #52 is addressed with a detailed `n.a.` explanation, contrasts to Category 2/3 and "Refuted", and examples that preserve existing category meanings.
- Curation thresholds, principles, and category meanings are unchanged.
- The Quarto book renders after content changes.

Phase 2 - content freshness acceptance:

- Frontend stack descriptions match current Vue 3, TypeScript, Vite, Bootstrap 5, and Bootstrap-Vue-Next usage.
- API/runtime descriptions no longer claim a single API script, rocker/tidyverse R 4.2.0, HAProxy, or NGINX rate limiting unless those claims are re-verified as current.
- Database runtime text matches current MySQL configuration.
- NDD Publications has a real analysis description in the style of neighboring analysis sections.
- Low-risk artifact-hygiene findings are either intentionally deferred or resolved with documented ignore/build-policy reasoning.

Phase 3 - first screenshot automation lane acceptance:

- A contributor can run a documented local command sequence to generate documentation screenshots.
- Generated screenshots are written as documentation assets, not failure artifacts.
- Each generated image has provenance metadata.
- The docs screenshot lane is explicitly isolated from default `npm run test:e2e` / bare default Playwright runs.
- The manifest can represent SPA routes, non-SPA/API URLs, authenticated roles, setup helpers, and action sequences.
- The first capture set covers public, analysis, authenticated reviewer, modal, and API surfaces.
- Verification reports missing references, missing generated files, and orphaned legacy screenshots.

Phase 4 - screenshot migration acceptance:

- Migrated Quarto references point to generated documentation assets.
- Generated overlays replace manual callouts for migrated screenshots.
- Legacy screenshots remain only where not yet migrated or where a screenshot is intentionally retained as non-generated.
- Shared images can be represented by multiple documentation references in provenance.
- Obsolete or orphaned legacy images are removed, archived, or explicitly deferred after section-level review.

Phase 5 - documentation design acceptance:

- Quarto remains the documentation framework.
- Image presentation is consistent.
- The docs visual language aligns with SysNDD's quiet clinical design direction.
- No design overhaul introduces marketing-style hero sections, decorative gradients, or card-heavy pages.
- Out-of-nav design documentation, including `10-visual-design-guide.md` and `11-admin-visual-review.md`, is either clearly referenced from durable docs or intentionally kept as planning/internal material.

Phase 6 - optional CI integration acceptance:

- Any CI screenshot workflow is manually triggerable before it is scheduled or wired into deployment.
- CI uploads generated screenshots as artifacts and builds Quarto from those artifacts when enabled.
- CI does not auto-commit generated screenshots in the first integration phase.
- Local generation remains the source of truth for debugging screenshot failures.
