## Code quality

### `app/src/composables/useAuth.ts` (W2)

Strong implementation. The contract documentation in the file header is exemplary — explicitly names the failure modes (corrupted localStorage, dangling token-or-user half-state, reactive drift between localStorage write and ref update) and documents the both-or-neither invariant in `syncFromStorage()`. `handle401()` is genuinely idempotent (the `wasLoggedIn` guard reads BOTH the refs AND localStorage, so a coalesced 401 storm can't cause a redundant navigation dispatch). The router import cycle is handled by relying on Vue's bundler-cycle resolution at call time, with a try/catch fallback for Vitest contexts that don't mount a router. The reactive wall-clock ref (`nowSecRef` driven by 1s setInterval) is a thoughtful detail that fixes a real cache-staleness bug in `isExpired`.

One small concern: the 1-second `setInterval` runs forever at module scope. Fine for an SPA tab but means tests that import `useAuth` need to be aware of the timer (they probably are; vitest's fake timers cover it).

### `app/src/plugins/axios.ts` (W2)

The interceptor delegation to `useAuth().handle401()` is correct: lazy import via `await import('@/composables/useAuth')` resolves the bundle-cache cycle, and the comment explains exactly why a top-level import would re-enter the cycle before module bindings are stable. The `isLoggingOut` guard is correctly described as belt-and-braces (since `handle401()` itself is idempotent), and the `__handled401` tag still works for concurrent 401 callers — the new tagged Error is created fresh on every concurrent rejection, so each `.catch()` block gets its own object with the tag attached.

### `app/src/api/entity.ts` and `app/src/api/user.ts` (W3)

Both follow the §5 typed-client contract exactly:
- Throw on non-2xx (no `{ok,data,error}` wrappers)
- Path params via `encodeURIComponent` (verified at every nested-route helper)
- Query params via `config.params` with proper spread merging (`{ ...config?.params, ...params }`)
- JSDoc on every export naming the R endpoint path, method, and source file line
- 404-as-no-data is a caller concern (documented in JSDoc)

`entity.ts` is particularly clean: handles both JSON and XLSX list formats via `listEntities` + `listEntitiesXlsx`, mirrors the R-handler-tolerant union for `EntityCreatePayload.review.phenotypes` (accepts both `{value}` and `{phenotype_id, modifier_id}` shapes), and the `nestedQuery()` helper correctly normalises the boolean-or-string `current_review` param to R's TRUE/FALSE.

`user.ts` correctly excludes `changePassword()` (it lives in `auth.ts` per the JSDoc note) — a clear ownership boundary that prevents drift.

### `app/src/views/review/Review.vue` and the new composables (W6)

The decomposition is clean. `Review.vue` is now an orchestration shell:
- 4 composables instantiated in `setup()` (`useReviewData`, `useReviewFilters`, `useReviewModals`, `useReviewActions`)
- 5 presentational components mounted in the template
- Methods retained on the component are thin glue (modal `$refs.show()`, the `loadReReviewData` orchestration that combines data fetch + filter pagination reset, plus pure formatting helpers).

Composable boundaries are cohesive: `useReviewData` owns table fetch + lookup lists + per-entity context; `useReviewFilters` owns filter/sort state with a computed `filteredItems`; `useReviewModals` owns modal open/close state; `useReviewActions` owns the approve/reject/submit mutations. Reactivity is preserved (the `watch(curation_selected, ...)` and `watch(filters.filteredItems, ...)` pattern correctly reacts to dependency changes).

Minor concern (not blocking): `Review.vue` lines 342-343 still contain `console.log(this.user.user_role[0])` and `console.log(this.curator_mode)` in the `mounted()` hook. These are the only `console.log` calls I noticed across the W6 surface; they should be removed in a follow-up cleanup.

### `app/docker/nginx/security-headers.conf` and `.planning/decisions/2026-04-25-csp-hsts-policy.md` (W1)

The CSP is correct. The directive at line 68:
- Uses `'sha256-ueWfB/sU1extuDGhe9jlQtX0uersTIhViOyPUGNpxlg='` for the inline JSON-LD block
- Drops `'unsafe-inline'` from `script-src` (verified by reading the directive)
- Retains `'unsafe-eval'` with the ADR justification
- Retains `'unsafe-inline'` for `style-src` with the ADR justification
- Includes `default-src 'none'`, `frame-ancestors 'self'`, `object-src 'none'`, `base-uri 'self'`, `connect-src 'self'`, `form-action 'self'`, `manifest-src 'self'`

The ADR adequately justifies retaining `'unsafe-eval'`: the three named vendor surfaces (Vue runtime template compiler reached via `ManageAbout.vue`, NGL Web Workers, markdown-it transforms) are all runtime-reachable, none are own-code, and none have a drop-in replacement that would survive a removal. The "alternatives considered" section is honest about the trade-off (NGL has no CSP-friendly replacement; nonce-based CSP requires SSR). I would have preferred to see the ADR also include a follow-up issue link for revisiting NGL or splitting the markdown-it transform path, but that's a polish observation, not a blocker.

### `app/scripts/audit-csp-violations.mjs` (W1)

Sound. The script walks `dist/` recursively, collects:
- Inline `<script>` tag bodies (excluding `src=` external scripts via the regex's negative lookahead `(?![^>]*\bsrc=)`), hashes them via sha256-base64, and prints the hash for inclusion in the directive
- Inline `style=""` attribute count (CSP cannot hash these; the audit reports them as informational counts)
- Eval-like calls (`eval(`, `new Function(`, `Function(`) in `.js`/`.mjs` files

Correctly skips `stats.html` (rollup-plugin-visualizer report) and `google*.html` (site-verification tokens) which exist in `dist/` but are not served. Exit code `1` if any of the three counters is non-zero — this is what the ADR claims.

The one limitation: the eval-like regex is purely textual, so it picks up any `Function(` substring including inside string literals. The ADR notes the audit reports 5 eval-like calls, which the directive accommodates by retaining `'unsafe-eval'`. False positives don't loosen the directive (they only document why the directive is loose), so this is fine.

### `app/playwright.config.ts` and `app/tests/e2e/fixtures/auth.ts` (Wave 0)

Config is correct. `globalSetup` re-seeds users between full runs (`a7729d16` fix) so password-update specs can mutate-and-restore without leaking cross-run state. `forbidOnly: !!process.env.CI`, `retries: 1` on CI, parallel workers — all sensible.

Auth fixture is well-engineered:
- Uses Playwright's request context (`request: APIRequestContext`) to call `/api/auth/authenticate` directly, bypassing the UI
- Bounded retry with backoff (5 attempts, 200ms × attempt) covers the password-update spec's mutation window — explicit and documented
- Hydrates the FULL `/api/auth/signin` payload into `localStorage.user`, not a minimal `{user_name, role}` stub. This matches the `useAuth.safeParseUser()` contract exactly (requires `exp: number[]` and `user_role: string[]`); a minimal payload would be silently rejected, leaving the SPA in a "logged-in but no user" half-state. The comment makes the intent explicit.
- Pre-acknowledges the disclaimer modal for every context (default + `loggedInAs`) via `addInitScript` writing to `localStorage` under the disclaimer pinia key. Idempotent.

### `docker-compose.playwright.yml` and `db/fixtures/playwright_users.sql` (Wave 0)

Fixture isolation is guaranteed:
- Volumes remapped to `sysndd_playwright_*` names so `docker compose -p playwright down -v` doesn't touch dev data
- SQL fixture sourced ONLY by `make playwright-stack` (verified: `grep -r 'playwright_users' db/migrations/` returns zero)
- Compose project name is `playwright` (explicit in the make target)

The compose overlay correctly omits `mysql-cron-backup` and `worker` (not needed for deep flows) and adds `mailpit` for fake SMTP (signup/password-reset flows that call `send_noreply_email`). Traefik Host rules expanded to allow `localhost`/`127.0.0.1` so Playwright can reach the stack on `http://localhost`.

The hashes in `playwright_users.sql` are sodium-scrypt format ($7$...) matching `api/core/security.R`'s verification path. Plaintext passwords are committed in `app/tests/e2e/fixtures/test-users.ts` as expected — these accounts only exist in the isolated playwright project.
