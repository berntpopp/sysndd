# v11.0 — Test Foundation & Structural Hardening

**Date:** 2026-04-11
**Milestone:** SysNDD v11.0
**Design doc for:** the first milestone after `v10.6`, addressing the findings in `docs/reviews/2026-04-11-codebase-review.md`
**Execution tooling:** superpowers skills (writing-plans → executing-plans → dispatching-parallel-agents + using-git-worktrees)
**Spec target:** 36 parallel worktree units across 5 sequential phase gates

## 1 — Summary

### 1.1 Problem statement

SysNDD is a mature research/clinical gene-disease database with a layered R/Plumber API and a Vue 3 SPA. Seventeen milestones have shipped. The 2026-04-11 consolidated review establishes an overall **5.0 / 10** scorecard across 15 categories and is blunt about the shape of the debt: *"It works, but each new feature is getting more expensive."*

The review identifies one live P0 (credentials in URL query strings), six Vue views over 1,400 LoC, three backend files over 1,200 LoC, a 971-line global-state bootstrap, a 630-line `legacy-wrappers.R` bridge that was never finished, in-memory async job state hidden behind a Traefik sticky-session cookie, TypeScript strictness effectively off (90% of `.vue` files still use plain `<script>`), and — most consequentially — a test pyramid that is upside-down: coverage threshold 40%, zero functional unit tests on the six largest views, only 5 of ~130 API endpoints with MSW handlers, empty `httptest2` fixture directories, no CI smoke test.

Section 6 of the review is explicit that testing is *"the single biggest multiplier for the rest of this plan"* — every structural refactor is unsafe without a safety net first. This spec turns that directive into an executable milestone.

### 1.2 Goal

Ship **v11.0 — Test Foundation & Structural Hardening**: put SysNDD into a state where the six "develop-against-confidence" scorecard categories score ≥8/10, without shipping a single refactor that isn't protected by a test written before it. Every worktree in this plan is either (a) building the safety net, (b) executing a refactor the net already protects, or (c) establishing the developer-environment hygiene that makes parallel work possible in the first place.

### 1.3 Scope

**In scope (v11.0):**

- P0 credentials-in-URL hotfix.
- Tier A testing infrastructure: MSW handler expansion, `httptest2` fixture recording, `skip_if_not_slow_tests` wiring, CI smoke test, `Sys.sleep` eviction.

- Backend structural refactors: delete `legacy-wrappers.R`, split the three god-files, extract `start_sysndd_api.R` into init modules, pagination sweep on 9 list endpoints.
- Frontend structural refactors: rewrite `ManageAnnotations.vue` and `ApproveReview.vue` as TypeScript, converge `ApproveStatus.vue` into a parameterized `ApprovalTableView` component, introduce `src/api/client.ts`, enable per-directory TypeScript strictness on router/api/types/auth composables.

## 2 — Architecture: phase gate model, worktree layout, merge protocol

### 2.1 Execution model

v11.0 is executed as **five sequential wave gates** (Phases A through E). Within each phase, work runs in parallel across independent git worktrees. A phase is complete only when every worktree in the phase has merged to `master` and CI is green.

**The key property:** the test-before-refactor gate is structural, not disciplinary. Phase D and E cannot open because their worktrees are created off `master` *after* Phase B+C have merged. There is no "forgot to wait for the test" failure mode because you cannot branch off something that doesn't exist yet.

### 2.3 Merge protocol

A phase has exactly one merge gate:

1. **Phase start** — all parallel worktrees for the phase are created off current `master` via `git worktree add` (the `make worktree-setup NAME=...` target from A7 is the canonical entry point).
2. **Parallel execution** — each worktree runs `make ci-local` locally before opening its PR.
3. **Per-PR review** — each unit is its own PR with its own review. For Phase C, reviews are **batch-reviewed** (all 11 PRs reviewed in one focused session via `superpowers:requesting-code-review`) to compress human bandwidth; see Risk 10 mitigation.
4. **Phase gate** — the phase is done only when every worktree's PR is merged to `master` **and** `make ci-local` on clean `master` is green **and**, for Phase B onward, the CI smoke test from B4 has completed successfully.

Detection of "phase done" is mechanical:

```bash
# Phase done when: every leaf branch in v11.0/phase-X/* is merged (remote + local)
git branch --list 'v11.0/phase-b/*' | wc -l          # must be 0
git ls-remote --heads origin 'v11.0/phase-b/*' | wc -l  # must be 0
make ci-local                                         # must be green on master
```

The execution session orchestrator (`superpowers:executing-plans`) runs this check after every PR merge; the next phase opens automatically when the check passes, subject to the human checkpoint in §2.7.

### 2.4 Intra-phase ownership rules

Parallel worktrees within the same phase can only land safely if they don't touch the same files. The wave decomposition in §3 was built around this constraint. Each worktree declares exclusive write ownership over specific paths; two worktrees in the same phase never appear in each other's ownership set.

Full ownership table for all 36 worktrees is in §3. Contested sections:

- **Phase D source-list edits** — D1, D2, D3, D4 all need to update the source list block in `api/start_sysndd_api.R`. **Rule:** those edits are made only inside a magic-commented region:

  ```r
  # --- function source list (v11.0) ---
  source("functions/...")
  # --- end source list ---
  ```

  Merge conflicts inside this block are trivially resolvable by alphabetizing, because source order matters only between layers (repositories before services — enforced by a test), not between files in the same layer.

- **Phase D `extract-bootstrap` (D6)** merges last, rebased against D1–D5's source-list edits. This is enforced by the execution session, not by branch protection.

- **Phase E `api-client-introduction` (E1)** merges first, because E3 (`first-client-migration`) and the view rewrites (E4/E5) all depend on the new `app/src/api/client.ts` module existing.

- **Phase E `converge-approve-status` (E6)** merges last in Phase E, because it reuses the pattern established by E5 (`rewrite-approve-review`) and their source files overlap conceptually.

### 2.5 Test-gate enforcement (the critical correctness check)

Every Phase D / Phase E refactor worktree must prove that the test protecting it was written *before* the refactor branch was cut, not alongside it. Layered enforcement:

**Layer 1: structural.** A Phase D/E worktree is created off `master` after Phase B+C have merged. By definition, the test already exists on `master` at the moment the branch is cut. A Phase D/E worktree that adds its own safety-net test is violating the plan.

**Layer 2: CI-enforced.** `scripts/verify-test-gate.sh` (added by Phase B worktree B4) runs on every Phase D/E PR and checks:

```bash
# For every *.spec.ts or test-*.R file modified in this PR:
#   - was the file created in this PR (allowed only for new unit/composable specs)?
#   - if pre-existing, does the diff touch anything other than an `it.todo` unpinning?
# If a pre-existing test file has non-.todo changes in a D/E PR: FAIL.
```

This is a ~40-line bash script; the logic is a `git merge-base` + `git log --follow` + `git diff` check.

**Layer 3: human checkpoint.** Before Phase D opens, the Phase B+C completion review asks explicitly: *"If I rewrote each of these views to do something subtly different, would this test catch it?"* If the answer is no for any test, a reinforcing Phase C worktree is opened before Phase D is unblocked.

### 2.6 The `it.todo` handshake

Every Phase C view functional test includes at least one `it.todo` pinned to current behavior:

```ts
// Phase C — written against pre-refactor ApproveReview.vue
describe('ApproveReview', () => {
  it('submits a review and shows the success toast', async () => { /* ... */ })
  it('blocks submission when classification is empty', async () => { /* ... */ })
  it.todo('TODO: verify the correct approver role appears in the audit trail')
})
```

Phase E's `rewrite-approve-review` (E5) unpins that `it.todo` as part of the rewrite commit:

```ts
// Phase E — after the rewrite
it('verify the correct approver role appears in the audit trail', async () => { /* ... */ })
```

If the Phase E rewriting agent cannot turn the `it.todo` into a passing assertion, the rewrite is incomplete and the PR is rejected. The handshake exists to defeat tautological tests (see Risk 1 in §5).

### 2.7 Human checkpoints

Three human review gates, aligned with natural phase boundaries:

1. **After Phase A** — quick scan that the hotfix and fast wins landed without surprises. ~15 minutes.
2. **After Phase B+C, before Phase D opens** — the most important checkpoint. Human reads every view spec file and answers: *"Are these tests a meaningful safety net for the refactor that's about to happen?"* This is where tautological tests get caught.
3. **After Phase D+E, before declaring v11.0 done** — human runs `make ci-local` on clean `master`, confirms exit criterion #18 (10-run or 7-runs-+-7-day flake-free streak).

No per-PR human gate beyond the standard review each PR gets.

### 2.8 Rollback protocol (expanded in §5.3)

- Phase A/B failures: `git revert` the one PR; open a `-v2` worktree; phase not "done" until replaced.
- Phase C failures: `git revert` the one PR; no downstream phase depends on it yet (the dependency is at phase-boundary, not per-commit).
- Phase D/E failures: `git revert` the refactor PR. The safety-net test is **not** reverted — it stays as proof of what the rewrite needs to meet. Re-attempt in a new worktree.

## 4 — Testing discipline, fixture hygiene, gate mechanics

### 4.1 Test layers

| Layer | Tool | Owns | Phases |
|---|---|---|---|
| Unit (pure) | `testthat` for R, `vitest` for TS | Pure functions, no DB, no HTTP, no Vue mounting | C10, C11 (composables); existing R unit tests |
| Integration (DB) | `testthat` + `with_test_db_transaction()` | R services, repositories, endpoint handlers hitting the real test DB inside a rolled-back transaction | C7, C8, C9 |
| Component (mounted) | `@vue/test-utils` + `vitest-environment-jsdom` + MSW | Six view functional tests — full mount, real children, MSW for HTTP | C1–C6 |
| Smoke (stack) | `curl` + `docker compose` + GitHub Actions | Boot the docker stack, hit `/api/health/ready`, verify migration apply, CI-only | B4 |

Out of scope for v11.0 (deferred to v11.1): Playwright E2E, Vitest Browser Mode, `openapi-typescript` contract types, `Schemathesis` fuzz testing, mutation testing.

### 4.2 TDD discipline inside every Phase D/E worktree

Every Phase D/E worktree runs this exact loop, adapted from `superpowers:test-driven-development`:

```
1. make worktree-setup NAME=phase-e/rewrite-approve-review
2. cd worktrees/phase-e/rewrite-approve-review
3. make install-dev         # idempotent on clean or pre-installed worktree
4. make doctor              # verify env
5. Run the protecting test against unchanged source — must be GREEN.
   Example: npx vitest run src/views/curate/ApproveReview.spec.ts
6. Begin the refactor/rewrite. The test WILL break at some point — that's the red phase.
7. Finish the rewrite; the test is GREEN again.
8. Unpin the it.todo and turn it into a passing assertion.
9. make ci-local            # must be green before opening the PR
10. Open PR via superpowers:requesting-code-review or equivalent.
```

**Rule:** a Phase D/E worktree may not merge a PR where the protecting Phase C test was *modified to make it pass*. The only legal change to a pre-existing spec file is unpinning an `it.todo` that was deliberately left pinned in Phase C.

`scripts/verify-test-gate.sh` (filled in by B4) enforces this on PR CI: it diffs every modified `*.spec.ts` or `test-*.R` file against `git merge-base master HEAD` and fails the PR if any non-`it.todo` line changed in a file that existed before the branch was cut.

### 4.3 MSW handler contract (Phase B1)

Every B1 handler follows one shape:

- Returns a response schema matching the real API shape (source of truth: `api/config/openapi/`).
- Includes both a 2xx success branch and at least one 4xx error branch reachable via a distinguishable request shape.
- Has the OpenAPI path as a code comment above the handler.

Enforcement:

- `scripts/verify-msw-against-openapi.sh` (new, lands with B1) greps every handler path and asserts it exists in the OpenAPI spec. Fake paths are caught.
- `onUnhandledRequest: 'error'` is flipped from `'warn'` as part of B1. Any un-mocked request in any Phase C test becomes a test failure, not a silent warning. This is the single biggest discipline improvement of Phase B.
- **Accepted residual risk (Risk 2):** handler response *bodies* can drift from real API shapes. The real fix (`openapi-typescript` compile-time checking) is deferred to v11.1. Mitigated by: human checkpoint #2 includes a 5-minute "curl the real API, diff against the handler" sanity check for each B1 handler.

### 4.4 `httptest2` fixture hygiene (Phase B2)

Three rules close the current "empty directory equals silently-skipping test" trap:

1. **Fixtures are committed, not `.gitkeep`-ed.** If `fixtures/pubmed/` or `fixtures/pubtator/` contains only `.gitkeep`, the helper fails the test loudly.
2. **Fixture provenance is documented.** `api/tests/testthat/fixtures/README.md` lists every fixture, when it was recorded, what API version it corresponds to, and the capture command needed to refresh it.
3. **Fixture refresh is an explicit make target.** `make refresh-fixtures` runs the capture commands against real APIs, invoked intentionally by a developer — never part of normal CI.

### 4.5 Default-on DB transaction rollback

Every `test-integration-*.R` file that doesn't need DDL is audited during Phase C (by C7/C8/C9, each in its subject area) and wrapped in `with_test_db_transaction()`. A CI grep check (added by B4's `verify-test-gate.sh` in an extended mode) asserts every integration test file opens with either `with_test_db_transaction` or a documented `skip_if_no_test_db()` exemption explaining why the rollback isn't usable.

### 4.6 Coverage threshold ratchet

```
Today          → lines 40 / functions 40 / branches 40 / statements 40
End of Phase B → 45 / 45 / 45 / 45  (bumped in B1's PR; MSW handlers unblock enough tests immediately)
End of Phase C → 55 / 55 / 55 / 55  (bumped in the last-merging Phase C PR)
End of v11.0   → unchanged at 55  (the refactor preserves coverage; the tests lifted it)
```

**Threshold changes always land with the tests that make them pass, never as a separate PR.**

Backend coverage (R) is measured via `covr::package_coverage()` in `make test-api` and the result is printed. Target ≥70%, **advisory only in v11.0**. Enforced floor lands in v11.1 once the realistic floor is known.

### 4.7 Flake-free streak definition

Exit criterion #18 asks for a flake-free streak. Definition:

- **Scope:** only triggered-by-push runs on `master` count, and only for the required CI jobs (`lint-api`, `lint-app`, `type-check`, `test-api`, `test-app`, `smoke-test`). Scheduled jobs (`slow-tests-nightly`) and jobs on non-v11.0 PR branches do not count toward or against the streak. This prevents v11.0 shipment from being held hostage to unrelated PR churn or scheduled nightly-job flakes.
- **Start condition:** the counter begins ticking the first time all required CI jobs go green on `master` *after* Phase E merges. Runs before Phase E is on `master` do not count.
- "Flake-free streak of 10 consecutive" means no red run on `master` in the streak window. Transient infra failures (docker pull retry, MySQL container slow-start) on `master` count as red and reset the counter — catching exactly that class of flake on the required path is part of the milestone's value.
- **Risk 4 exit ramp:** if after 2 weeks of Phase E completion the counter has not reached 10, the criterion relaxes to "7 consecutive green runs on `master` plus zero red runs on `master` in the preceding 7-day window." Still master-only. This ramp is explicit, not a silent downgrade.

The counter is tracked manually in v11.0 (a checklist in the v11.0-exit PR). An automated tracker in `.ci-flake-counter.md` is a v11.1 follow-up.

### 4.8 Local developer environment (Phase A7 deliverable)

The `make install-dev` + `make doctor` + `docs/DEVELOPMENT.md` bundle owned by A7 solves a concrete parallel-work problem: a worktree agent arriving at a fresh checkout must be able to go from `git worktree add` to running tests in two commands.

**Gaps identified (all closed by A7):**

| Gap | Impact | Fix |
|---|---|---|
| No aggregate `make install-dev` | Agents discover `install-api` + `install-app` + DB bootstrap manually | New `install-dev` target |
| No `make doctor` | No way to verify "env is ready" without running tests and inferring from failures | New `doctor` target |
| No `.nvmrc` or `engines.node` pin | Silent Node-version drift from CI | `app/.nvmrc` pins to CI's Node major |
| No `CONTRIBUTING.md` or `docs/DEVELOPMENT.md` | New contributor has no "start here" doc; `CLAUDE.md` is agent-facing, not human-facing | Both files created |
| System deps undocumented | Docker, git ≥2.5 (worktrees), make, optional jq/gh/mysql-client not enumerated anywhere | Listed in `docs/DEVELOPMENT.md` §1 |
| R dev package presence unverified | `covr`, `lintr`, `styler`, `httptest2`, `callr`, `mockery` may or may not be in the lockfile | A7 audits and adds to `renv.lock` |

**Best-practices applied:**

| Area | Practice | Applied |
|---|---|---|
| R package mgmt | `renv` with committed lockfile, non-interactive restore | Already in place; A7 verifies dev packages present |
| Node package mgmt | `npm ci` (not `npm install`) for reproducibility; Node pinned via `.nvmrc` | Already in CI; A7 adds `.nvmrc` for parity |

**Worktree node_modules optimization:** *skipped in v11.0.* Symlinking `node_modules/` from the main worktree is fiddly (lock-file edits, broken symlinks) and the wall-time savings don't compound beyond the per-worktree setup cost. Noted as a v11.1 option if parallel-work friction proves to be a real bottleneck.

**Cross-platform support:** A7's PR runs `make doctor` across Ubuntu and macOS via GitHub Actions matrix. `make doctor` must pass on both before A7 is mergeable.

## 5 — Risks, failure modes, rollback

### 5.1 Top 10 risks

| # | Risk | Prob. | Impact | Phase |
|---|---|---|---|---|
| 1 | Phase C tests are tautological — written against current source, they pin current behavior *including its bugs*, and the Phase E rewrite "passes" by reproducing the bugs | High | High | C → E |
| 2 | MSW handler drift from real API — handlers return the shape the agent guessed, not the shape the API actually returns; view tests pass but production breaks | Med | High | B1 → C → E |
| 3 | Merge conflicts in Phase D despite ownership rules — `start_sysndd_api.R` source-list edits collide across D1/D2/D3/D6 | Med | Med | D |
| 4 | The 10-green-runs gate is never achieved — infra flakes, PR-open churn, or scheduled jobs (even though excluded from the count) destabilize `master` | Med | Med | End of v11.0 |
| 5 | R renv drift — Phase B2 or B4 adds a package, the lockfile includes host-OS-specific paths | Med | Med | B |
| 6 | Phase C takes 2× the estimated time — 11 worktrees × 1 test per target is a lot of review bandwidth, not a lot of agent execution | Med | Low | C |
| 7 | A view rewrite silently changes user-visible behavior the Phase C test didn't cover (e.g. a filter default changes) | Med | Med | E |
| 8 | Default-on `with_test_db_transaction` breaks tests that relied on implicit commit | Low | Med | C |
| 9 | A7's `make doctor` passes on agent machines but fails on a real contributor's machine (different OS, different Node, different R) | Low | Low | A |
| 10 | Parallel agent output review becomes the bottleneck — 36 PRs in a month is 1 per day on average, but Phase C's 11 in one wave is a spike | Med | Low | C, E |

### 5.2 Mitigations

**Risk 1 — Tautological tests.** The most dangerous risk. Three layers of defense:

- **Layer 1 — the `it.todo` handshake.** Every Phase C view test includes at least one `it.todo` describing a behavior the test knows it cannot verify against unchanged source. Phase E unpins that `it.todo` as part of the rewrite. Rewrites that cannot turn the `it.todo` into a passing assertion are rejected.
- **Layer 2 — human checkpoint #2.** Reviewer reads every view spec before Phase D opens and answers: *"If I rewrote this view to do something subtly different, would this test catch it?"* No → a reinforcing worktree opens in Phase C before Phase D is unblocked.
- **Layer 3 — mutation testing.** `stryker-js` on view specs would automatically detect tautological tests. **Deferred to v11.1.** Noisy on Vue SFCs; not free. But exactly the tool that prevents recurrence.

**Risk 2 — MSW handler drift.** Phase B1 handlers reference their OpenAPI path as a comment and are checked by `scripts/verify-msw-against-openapi.sh`. Body-shape drift remains a residual risk; mitigated at human checkpoint #2 by a 5-minute "curl the real API, diff against the handler" sanity check per handler. Compile-time contract enforcement (`openapi-typescript`) is v11.1 work and eliminates this risk class entirely.

**Risk 3 — Phase D source-list conflicts.** Three defenses:
- D6 (`extract-bootstrap`) explicitly merges last.
- D1/D2/D3/D4 edit `api/start_sysndd_api.R` only within a magic-commented source-list block; conflicts inside the block are trivially resolvable by alphabetization (source order matters only between layers, not within a layer — enforced by a test).
- A source-list test (existing or new) asserts repositories source before services.

**Risk 4 — 10-green-runs gate never reached.** Two layers:
- The counter is **master-only** and excludes scheduled jobs (`slow-tests-nightly`). Only triggered-by-push runs on `master` for the required CI jobs count. This is the narrowed definition in §4.7 and exit criterion #18 — it prevents unrelated PR churn and nightly-job noise from holding v11.0 hostage.
- **Exit ramp:** if after 2 weeks of Phase E completion the counter hasn't reached 10, the criterion relaxes to "7 consecutive green on `master` plus zero red on `master` in the preceding 7-day window." Still master-only. Still excludes scheduled jobs. This is documented in exit criterion #18 as an explicit allowed downgrade.

**Risk 5 — renv drift.** Every Phase B worktree adding an R package commits the `renv.lock` update in the same commit as the package use, and runs `renv::status()` locally before PR. A7's `make doctor` includes `renv::status() | grep -q "synchronized"`.

**Risk 6 — Phase C takes longer than expected.** Pure timeline risk. Mitigation: Phase C does not block Phase D on a single straggler — if 9 of 11 worktrees merge on schedule, Phase D may start on the merged-9 subset provided the stragglers are for endpoints/composables that D does not touch. If the stragglers are view tests, Phase D waits.

**Risk 7 — silent UX regression.** For each view rewrite PR (E4, E5, E6), the rewriting agent captures before/after screenshots of the happy path in dev mode and attaches them to the PR. No automated visual regression yet (Tier C / Playwright / v11.1) — but side-by-side screenshot review is cheap and catches the obvious cases. `superpowers:requesting-code-review` instructions for Phase E include this requirement.

**Risk 8 — `with_test_db_transaction` default-on breaks existing tests.** The Phase C worktree that flips this default runs the full `make test-api` suite in its own worktree and fixes any broken test in the same PR. If more than 3 tests break, the worktree is split and the transaction-default flip ships as a separate sequenced PR within Phase C.

**Risk 9 — cross-platform `make doctor`.** A7's PR includes a GitHub Actions matrix job running `make doctor` on `ubuntu-latest` and `macos-latest`. A7 is not mergeable until both pass.

**Risk 10 — review bandwidth bottleneck.** Phase C's 11 worktrees are batch-reviewed in a single focused session via `superpowers:requesting-code-review`. The review question is one ("are these tests a meaningful safety net?"), asked once across all 11 PR refs. Individual PRs get `LGTM` after the batch review confirms the whole wave is sound. This compresses 11 review sessions into 1.

### 5.3 Rollback playbook

| Situation | Response |
|---|---|
| Phase A/B worktree PR lands and breaks `master` | `git revert` the single PR; re-open on a `-v2` branch; phase is not "done" until replaced |
| Phase C test merges, then flags as flaky in subsequent runs | Do not revert — mark the spec as `.skip` with an issue reference; open a reinforcing worktree later in the same phase to fix or replace the test |
| Phase D refactor PR lands and `make test-api` goes red on `master` | `git revert` the refactor PR immediately. **Do not revert the protecting Phase C test** — it stays as proof of what the refactor needs to meet. Re-attempt in a new worktree |
| Phase E view rewrite PR lands and the view breaks in dev | Same as above — revert the rewrite, keep the test, re-attempt |
| Two Phase D PRs merge close in time and cause a transitive break (each is green in isolation, both together is red) | `git revert` the later one; document the coupling in the revert commit message; re-attempt the later one with the coupling addressed |
| A Phase C view test turns out to be tautological post-merge | Do not revert the test. Open a Phase C.5 reinforcing worktree that adds a stronger assertion and deletes the weak one in the same PR. This is test refactoring, not test removal |
| CI smoke test (B4) starts failing due to a docker base image change | Pin the base image version in the relevant Dockerfile; treat as an infra fix, don't revert v11.0 work |

### 5.4 Exit ramps

Each phase has a point at which, if convergence hasn't happened, the plan explicitly downshifts rather than failing silently.

- **Phase A doesn't converge in 1 week** → someone is blocked on a hotfix review. Escalate to a synchronous review session rather than queuing.
- **Phase B doesn't converge in 1.5 weeks** → MSW handler set is larger than expected. Acceptable to defer B1's handlers for views C5/C6 into Phase C as an exception to the ownership rule, since nothing else blocks on them.
- **Phase C doesn't converge in 3 weeks** → real signal that view tests are hard. Pause Phase C; run a retrospective on *why* they're hard; reduce the per-view test scope to "happy path only"; reopen Phase C with the reduced scope. Ship with smaller test coverage rather than ship without tests.
- **Phase D doesn't converge in 2 weeks** → source-list conflicts worse than estimated. Convert D1/D2/D3 into strict sequential order and rerun.
- **Phase E doesn't converge in 3 weeks** → the view rewrites are fighting the tests. Signal that the v11.0 refactor scope was too ambitious. Defer E4/E5 to v11.0.5, close v11.0 on what did land. The coverage-threshold bumps still stand; the refactor scope shrinks.

Each ramp is an explicit option, not a failure. The plan is "go fast with guardrails"; the guardrails are the ramps.

### 5.5 Explicitly NOT risks

- **36 worktrees sounds like too many.** Mean phase is 7 worktrees; max (Phase C) is 11. `git worktree list` handles hundreds without issue.
- **Parallel agents stepping on each other.** Physically impossible given §2.4's ownership table — they cannot write the same file from different worktrees.
- **Test-gate script complexity.** ~40 lines of bash (`git merge-base` + `git log --follow` + `git diff`).
- **TS strictness sweep breaking the global build.** Per-directory `tsconfig.*.json` extends the global one; the global stays permissive; only scoped files get strictness. Global build unaffected.
- **`legacy-wrappers.R` deletion breaking a caller we missed.** Covered by C7/C8/C9's endpoint test batches, which exercise every write endpoint that currently routes through the wrappers.

### 5.6 A considered-and-rejected mitigation

**AI pre-review of PRs.** Tempting as a way to cut human review bandwidth, but rejected because: (a) it doubles tool-call budget, (b) it makes human review more cursory ("the AI already looked"), and (c) the batch-review pattern in Risk 10's mitigation solves the same problem more simply. If human review time really is the bottleneck, AI pre-review is a v11.1 process improvement — not scope for this spec.


Seven sessions total after this one. Each execution session is focused on a single phase with a natural stopping point.

### 6.2 What the `writing-plans` session produces

For each phase, one plan document at `.plans/v11.0/phase-<x>.md` containing:

- **Prerequisites check** — what must be green on `master` before the phase opens.
- **Worktree manifest** — exact branches, owners, ownership table rows from §3.
- **Per-worktree task spec** — goal, acceptance, file ownership, test-gate reference, the TDD loop from §4.2.
- **Parallel dispatch block** — explicit list of which worktrees can run concurrently in the `superpowers:dispatching-parallel-agents` sense.
- **Checkpoint** — which of the three §2.7 human reviews happens at this phase's close.
- **Phase gate check** — the exact commands the session runs to decide "phase done."

**Recommendation for the writing-plans session:** produce all five phase plans in one pass. Having all five in hand means execution sessions can start in sequence without returning to writing-plans between them.

### 6.3 Timeline expectations

No week-counts (per project instructions on avoiding estimates). Structural expectation:

- Phase A is the shortest (7 small, independent units, no test dependency).
- Phase C is the longest (11 worktrees each producing new tests from scratch, plus batch review).
- Phase D is medium (6 backend units, one sequenced last).
- Phase E is medium-long (7 frontend units, two sequenced; view rewrites and the auth/session consolidation are the hardest individual units in the whole plan).
- Phase B is short-medium (5 infra units, mostly independent, some conceptual design in B1 and B4).

Wall-clock timing matters less than **gate order** — v11.0 is done when Phase E's gate passes, and each gate's duration is bounded by the §5.4 exit ramps.

### 6.4 Success looks like this

Once v11.0 ships, a SysNDD contributor (human or agent) opening curation-workflow work for v11.1 sees:

- `make install-dev` works from a clean clone in one command.
- `vitest --watch` gives sub-second feedback on a view change, using real MSW handlers that match the actual API.
- `make test-api` finishes fast on a dev machine, hits a real test DB, rolls back every transaction.
- The biggest Vue view is ~700 lines, built on typed composables — no 2,159-line monolith.
- `legacy-wrappers.R` does not exist; every write endpoint routes directly through a typed service.
- A broken deployment is caught by the CI smoke test before the PR merges.
- Credentials are never in a URL.
- The scorecard shows 8/10 across the six confidence categories; v11.1 roadmap is free to focus on operational hardening (TLS, observability, durable jobs) instead of playing catch-up on debt.
